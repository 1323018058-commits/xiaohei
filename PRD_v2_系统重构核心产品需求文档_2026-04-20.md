# ProfitLens / 小黑 ERP
# 系统重构核心产品需求文档（PRD v2）与架构蓝图

**文档版本**：v2.0  
**日期**：2026-04-20  
**文档用途**：供产品、架构、前端、后端、测试、外包/重构团队从零重建系统使用  
**产品定位**：面向 Takealot 卖家的垂直 ERP / 运营执行系统  
**重构方式**：放弃当前代码库实现，从零重构，但保留当前系统沉淀出的业务真相  
**核心原则**：后端给真相，前端做表达，状态机驱动业务，任务流驱动长耗时动作

# 0. 执行摘要

ProfitLens / 小黑 ERP 不是单点工具，而是一套围绕 **“选品 → 铺货 → 竞价 → 履约 → 财务”** 的卖家执行系统。

它的真正价值不在于“展示后台页面”，而在于帮助 Takealot 卖家完成完整运营闭环：

1. 从外部商品源发现可售商品  
2. 通过 AI 改写和模板填充快速生成铺货任务  
3. 在 Takealot 平台上自动竞争 BuyBox，同时避免价格失控  
4. 成交后把订单、采购、PO、物流、仓库动作串成可执行履约流程  
5. 最终形成利润、费用、钱包流水和经营状态的可审计结果

正式重构后的系统应当是一个：

- **模块化单体系统（Modular Monolith）**
- 以 **Store / Product / ListingJob / BidProduct / Order / PO** 为核心对象
- 以 **Task / Status / Stage / ui_meta** 为业务执行骨架
- 以 **Takealot / 1688 / Amazon / CNExpress** 为外部平台连接层
- 以 **Next.js 前端 + Python 后端 + PostgreSQL 主库 + Redis 运行态** 为推荐技术基线

一句话定位：

> ProfitLens 是一个面向 Takealot 卖家的垂直运营 ERP，通过商品情报、自动铺货、自动竞价、履约合单和财务快照，把卖家的日常运营动作变成一个可执行、可审计、可放大的交易系统。

# 1. 全局系统大纲（System Overview）

## 1.1 产品使命

帮助 Takealot 卖家在一个系统中完成：

- 店铺接入与健康监控
- 商品情报沉淀
- 自动选品与利润判断
- 自动铺货与审核跟进
- 自动竞价与价格护栏
- 履约合单、采购、物流、仓库协同
- 财务费用、钱包流水、利润快照
- 管理后台、订阅、用户与权限管理

---

## 1.2 核心商业价值

| 价值 | 说明 |
|---|---|
| 提高选品效率 | 沉淀 Takealot 商品情报，支持利润筛选和历史回看 |
| 提高铺货效率 | 从链接或关键词批量生成铺货任务，减少人工制表 |
| 提高价格竞争力 | 自动追踪 BuyBox，在安全区间内执行调价 |
| 降低履约混乱 | 将订单、PO、物流、仓库动作整合到一个履约工作台 |
| 提高财务透明度 | 将销售、采购、物流、税费、钱包流水转成利润快照 |
| 支撑商用化 | 具备登录、限流、会话、任务、审计、健康度等基础设施 |

---

## 1.3 主要功能模块

1. 账户与订阅（Auth / Subscription）
2. 店铺管理（Store Management）
3. 商品情报与选品库（Selection Library / Product Intelligence）
4. 自动选品（Auto Selection / Product DB）
5. 自动铺货（Listing / Dropship）
6. 自动竞价（AutoBid）
7. 履约中心（Fulfillment / PO Workbench）
8. 财务中心（Finance）
9. 仓库与物流协同（Warehouse / CNExpress）
10. 浏览器扩展（Extension）
11. 管理后台（Admin）

---

## 1.4 核心业务主链

### 主链一：商品生成链

```text
外部商品源
  → 抓取 / 图搜 / 匹配
  → AI 改写
  → 类目选择
  → 填写 Takealot loadsheet
  → 提交审核
  → 回写 submission / review 状态
```

### 主链二：价格竞争链

```text
同步店铺商品
  → 获取 BuyBox
  → 读取 SKU 全局底价
  → 计算理想价 buybox - 1
  → 应用 floor / ceiling 双保险
  → 执行调价
  → 写入 bid_log
```

### 主链三：履约执行链

```text
官方订单导入
  → 订单池
  → 按相同目的地仓合单
  → 创建 PO
  → PO 内逐单补快递单号 / 采购信息
  → 物流 / 仓库动作
  → 完成或异常
```

### 主链四：财务结算链

```text
销售收入
  + 采购成本
  + 头程 / 尾程
  + 佣金 / VAT / 关税
  + 仓库 / 嘉鸿费用
  → 利润快照
  → 财务中心
```

# 2. 产品目标与非目标

## 2.1 产品目标

### G1. 建立可执行的商品生成链

系统必须支持从商品链接或关键词生成上架任务，并可追踪审核结果。

### G2. 建立受保护的自动竞价链

系统必须支持自动调价，但所有调价必须受人工底价和平台安全上限保护。

### G3. 建立可审计的履约执行链

系统必须支持订单池、同仓合单、PO、逐单补号、仓库与物流动作。

### G4. 建立利润与费用可追溯链

系统必须能把销售、采购、物流、费用和利润结果显式化。

### G5. 建立可商用的基础设施

系统必须具备认证、限流、会话、幂等、任务、日志、监控、健康检查等能力。

---

## 2.2 非目标

以下内容不作为第一期重构目标：

- 多平台通用 ERP
- 原生移动端
- 复杂组织/多租户架构
- 全自动“选品 → 铺货 → 竞价”审批流水线
- 通用 BI 平台
- 微服务化拆分

# 4. 业务域拆分（Domain Blueprint）

## 4.1 Auth / Subscription

### 目标

提供稳定的登录、激活、订阅和权限控制。

### 核心能力

- 注册
- 登录
- 注销
- 当前用户
- 激活码激活
- 订阅可用性判断
- Redis Session Cache
- Lua 限流
- 登录并发 admission gate
- Split-Auth 物理隔离

### 核心规则

- 前端登录态必须以 `/api/auth/me` 为最终真相
- 用户是否可使用系统由 `can_use_core_features` 决定
- 登录过载时应返回明确 admission / rate limit 错误

---

## 4.2 Store Management

### 目标

管理 Takealot 店铺资产、API 凭证、同步状态与健康度。

### 核心能力

- 店铺列表
- 新增店铺
- 编辑店铺
- 删除店铺
- API Key 校验
- 店铺同步
- Webhook 配置
- 健康评分

### 核心规则

- `api_key_status` 是后端持久化字段
- 同步成功/失败必须回写状态
- 店铺同步属于异步高延迟动作

---

## 4.3 Product Intelligence / Selection Library

### 目标

沉淀 Takealot 商品情报，支撑选品和后续自动化。

### 核心能力

- 商品标题
- 品牌
- 类目
- 图片
- 价格区间
- 评分与评论
- 交期
- 库存
- 搜索、筛选、分页
- 商品注释

---

## 4.4 Auto Selection

### 目标

从商品情报库中筛选候选商品，并计算利润与风险。

### 核心能力

- 发起抓取任务
- 查看任务进度
- 中间结果预览
- 利润试算
- 导出报告
- 查看历史批次

### 核心规则

- 自动选品只负责筛选，不自动触发铺货
- 失败原因必须结构化记录

---

## 4.5 Listing / Dropship

### 目标

把链接或关键词结果转化为 Takealot 可审核的上架任务。

### 子模块

#### 链接直铺

```text
输入外部商品链接
  → 抓商品详情
  → AI 改写
  → 选模板
  → 填 loadsheet
  → 提交 Takealot
```

#### 关键词采买

```text
输入关键词
  → 抓 Amazon seeds
  → 1688 图搜匹配
  → 相似度过滤
  → AI 改写
  → 提交审核
```

### 核心规则

- Brand 固定为 `Generic`
- 初始售价留空
- 不和自动竞价强耦合
- 不和自动选品强耦合

---

## 4.6 AutoBid

### 目标

在安全护栏内自动竞争 BuyBox。

### 核心能力

- 同步在售商品
- 获取 BuyBox
- 导入 SKU 底价
- 自动调价
- 调价日志
- 店铺级启停
- 运行状态监控

### 核心规则

- 无人工底价 → 禁止竞价
- 理想价 = `BuyBox - 1 ZAR`
- 低于底价 → 锁定底价
- 高于 ceiling → 锁定 ceiling
- ceiling = `synced_base_price * 1.10`
- 废除旧 `当前售价 × 90%` 自动底价逻辑

---

## 4.7 Fulfillment / PO Workbench

### 目标

将官方订单、内部 PO、采购物流、仓库动作串成履约执行链。

### 核心流程

```text
订单池
  → 勾选同仓订单
  → 合并 PO
  → 打开 PO 对账弹窗
  → 逐单填写快递单号 / 备注
  → 推进物流 / 仓库流程
```

### 核心规则

- 禁止跨仓库合单
- PO 是同目的地仓订单的容器
- PO 内每个订单都需要单独补号
- `PO_Manager` 是 PO 写路径唯一 owner
- 已完成 / 已取消对象默认只读

---

## 4.8 Finance

### 目标

汇总收入、采购、物流、税费和利润。

### 核心能力

- 钱包
- 钱包流水
- 利润快照
- 嘉鸿费用
- 对账视图

---

## 4.9 Warehouse / CNExpress

### 目标

支持仓库与嘉鸿物流协同。

### 核心能力

- 仓库动作
- 到仓
- 打包
- 贴标
- 出库批次
- 嘉鸿订单
- 嘉鸿钱包
- 物流跟踪

---

## 4.10 Extension / Admin

### Extension

用于外部页面上的即时利润计算和快速铺货动作。

### Admin

用于用户、license、系统健康、分类审核等站点级管理。

# 6. 后端核心业务逻辑与 API 契约

## 6.1 后端总体架构

后端由五层组成：

1. Flask 路由层
2. Domain Service / Service 层
3. Worker / Task 执行层
4. 外部平台连接器
5. Redis + DB 运行态基础设施

它是一个典型的：

```text
API + Worker + 外部平台适配器 + 双数据库
```

而不是单纯的 CRUD 后端。

## 6.3 隐性业务规则

### 利润公式（跨境利润）

利润并不是简单的 `售价 - 成本`，而是：

```text
售价(CNY)
  - 前段成本（采购 + PO 费 + 贴单费 + 运费 + 关税 + 进口 VAT）
  - 佣金
  - 佣金 VAT
  - 尾程配送费
  - 尾程 VAT
  - 回款损耗
= 利润
```

### 自动竞价护栏

```text
ideal_price = buybox - 1 ZAR
ceiling = synced_base_price * 1.10

if no manual_floor_price:
    skip
elif ideal_price < manual_floor_price:
    final = manual_floor_price
elif ideal_price > ceiling:
    final = ceiling
else:
    final = ideal_price
```

### 铺货强制规则

- Brand 固定为 `Generic`
- 初始价格留空
- 模块解耦，不自动串成审批流水线

### 履约核心规则

- `PO_Manager` 是唯一写路径 owner
- `FULFILLED / CANCELLED` 默认只读
- 前端只消费 `status / stage / ui_meta`
- 官方导入与内部刷新分离

### 身份与会话规则

- Redis Session Cache
- Redis Lua rate-limit
- 并发登录 admission gate
- `/api/auth/me` 是最终登录态真相
- Split-Auth 把 auth 请求和业务请求物理隔离

# 12. 一句话结论

不要把 ProfitLens 当成“管理后台”来重构。  
它应该被当成一个 **面向 Takealot 卖家的执行型 ERP** 来重构。

真正的系统主线是：

```text
Store
  → Product Intelligence
  → Listing / Dropship
  → AutoBid
  → Fulfillment
  → Finance
```

其中最重要的商业护城河是：

1. 铺货自动化
2. 自动竞价护栏
3. 履约合单与 PO 工作台
4. 跨境利润公式
5. 对 Takealot / 1688 / Amazon / CNExpress 的实战接入经验

如果重构团队只照着旧页面做，会失败。  
如果围绕这里定义的业务真相、状态机和任务流来做，就能重建出一个真正可商用的 Seller ERP。
