# 压测执行口径冻结

## 1. 目标

本文件继续下沉 `13_压测脚本与造数方案.md`，把压测从“有方案”推进到“执行口径冻结”，避免后续在脚本组织、造数职责、执行批次和通过标准上反复改口。

冻结范围：

- 目录结构
- `Locust` 用户类型
- 造数职责
- 执行批次
- 阈值与阻断标准
- 失败记录模板
- 上线前复验口径

---

## 2. 总原则

- 压测工具固定为 `Locust`
- 先冻结执行口径，再写脚本细节
- 先保 `P0` 场景，再扩展 `P1`
- 压测结果采用分级通过：
  - `可上线`
  - `理想通过`

## 4. Locust 用户类型冻结

### 4.1 鉴权与控制面

- `AdminUser`
- `OperatorUser`
- `AdminReadUser`
- `OperatorReadUser`

### 4.2 任务与同步

- `SyncTriggerUser`
- `RecalcTriggerUser`
- `BidBatchTriggerUser`

### 4.3 高危写场景

- `AdminWriteUser`
- `FulfillmentWriteUser`
- `BiddingWriteUser`

冻结规则：

- 新增用户类型必须能映射到现有场景，否则不随意扩充
- 同一用户类型的职责必须单一，不混写多个业务域动作

## 5. 造数职责冻结

### 5.1 `seed_users.py`

负责：

- 用户、角色、订阅、功能开关、基础会话样本

### 5.2 `seed_stores.py`

负责：

- 店铺、凭证状态、同步状态、Webhook 入口样本

### 5.3 `seed_fulfillment.py`

负责：

- 订单、订单项、PO、Shipment、异常样本

### 5.4 `seed_bidding.py`

负责：

- 竞价商品、底价、策略、运行记录、护栏阻断样本

### 5.5 `seed_tasks_and_audits.py`

负责：

- 任务状态分布、事件流、死信样本、审计样本

### 5.6 `seed_finance.py`

负责：

冻结规则：

- 不允许在一个脚本里混灌多个域的主数据
- 压测前先造控制面，再造主链业务，再造运行面样本

## 6. 执行批次冻结

### Batch A：容量预估演练

范围：

- 场景 A：登录与基础鉴权
- 场景 B：列表查询高并发

目标：

- 找出最早的连接池、读接口或限流瓶颈

### Batch B：单域压测

范围：

- Store
- Fulfillment
- Bidding
- Finance（只跑读侧与白名单重算）

目标：

- 验证每个域独立稳定性

### Batch C：P0 全链路压测

范围：

- 场景 A + B + C + D

目标：

- 验证主站主链是否可支撑首发

### Batch D：异常隔离压测

范围：

- 场景 E
- 并发叠加部分读请求

目标：

- 验证外部失败不拖垮主站

### Batch E：上线前复验

范围：

- Runbook 彩排后的小规模复验
- 关键路径 smoke + 压力复验

目标：

- 确认发布版本仍满足上线门槛

## 7. 分级通过标准冻结

### 7.1 可上线

满足以下条件即可视为“可上线”：

- 登录与 `/api/auth/me` 稳定
- 核心读接口错误率受控
- 核心写接口无重复提交失控
- 长任务创建可返回稳定 `task_id`
- 外部失败不会拖垮主站
- 队列积压可观测
- 高危写接口审计完整

### 7.2 理想通过

在“可上线”基础上，进一步满足：

- P95 / P99 响应时间达到目标
- 数据库热表扫描受控
- 队列无持续积压
- 重试与死信规模保持在预期范围内
- `P1` 读侧与受限能力也稳定

### 7.3 阻断上线

满足任一条件即阻断上线：

- 重复调价
- 重复财务入账
- 重复 PO
- `/api/auth/me` 不稳定
- 5xx 持续升高
- 队列无法消费且恢复失败
- 审计缺失

## 8. 失败记录与报告输出冻结

### 8.1 失败记录模板

```text
[压测失败记录]
- 时间段：
- 批次：
- 场景编号：
- 失败接口 / 任务类型：
- request_id / task_id：
- error_code：
- 关键指标快照：
- 初步原因：
- 是否阻断上线：是 / 否
- 处置动作：
```

### 8.2 报告输出

每轮压测必须产出：

- 原始报告
- 摘要报告
- 失败列表
- 风险结论
- 是否达到“可上线 / 理想通过”

---

## 9. 上线前复验口径冻结

上线前复验必须同时覆盖：

- Runbook 彩排
- Auth / Admin smoke
- Store 同步任务创建
- Fulfillment 主链
- Bidding 读侧与受控写侧
- Finance 读侧与白名单重算入口
- 指标复核：错误率、队列、连接池、审计完整性

冻结规则：

- 未完成复验，不得宣布准备上线
- 复验结果必须用“可上线 / 理想通过 / 阻断上线”三档表达

---

## 10. 关键冻结点

- `Locust` 为唯一主工具
- 脚本目录与造数职责固定
- 执行批次固定为 A~E
- 通过标准采用三档：`可上线 / 理想通过 / 阻断上线`
- `Finance` 复验仍遵守白名单重算边界
