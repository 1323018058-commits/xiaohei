# 压测脚本与造数方案

## 1. 目标

本文件把 `07_压测计划.md` 继续拆解为可执行的压测准备方案，覆盖：

- 环境准备
- 造数方案
- `Locust` 场景脚本清单
- 观测项与看板
- 失败记录模板
- 执行批次与上线前复验

本轮默认以 `Locust` 作为首选压测工具。

---

## 2. 为什么选择 Locust

- 支持 Python 编排复杂业务流程
- 适合 ERP 这类多步骤接口链路压测
- 更容易表达登录、鉴权、任务创建、轮询、异常注入等行为
- 后续更容易把压测脚本和造数脚本放在同一语言栈内维护

## 4. 环境准备清单

### 4.1 基础环境

- PostgreSQL：接近生产配置
- Redis / 队列：开启队列长度与消费速率监控
- Worker：至少提供与首发预期相近的并发数
- API 服务：开启结构化日志、接口耗时日志、错误日志
- 监控系统：可观测应用、数据库、队列、任务

### 4.2 外部依赖策略

- Takealot：使用 sandbox 或可控 mock
- CNExpress：使用超时 / 异常可控 mock
- 1688 / Amazon / AI：使用失败注入桩，不阻断主站

### 4.3 配置要求

- 开启登录限流
- 开启 API 限流
- 开启错误脱敏
- 保留 request_id、task_id、error_code
- Dashboard 读侧必须接汇总源，不直扫热明细

### 4.4 验收口径

- 环境日志齐全
- 监控指标可实时看到
- 模拟外部失败不会直接打挂主站

### 6.2 场景 A：登录与基础鉴权

目标：

- 验证 `login` 与 `/api/auth/me` 稳定

Locust 用户类型：

- `AdminUser`
- `OperatorUser`

脚本职责：

- 登录获取会话
- 周期性请求 `/api/auth/me`
- 记录 401 / 429 / 5xx 比例

关键接口：

- `POST /api/auth/login`
- `GET /api/auth/me`

通过标准：

- 登录错误率 `< 1%`
- `/api/auth/me` 稳定
- 限流命中时返回 `RATE_LIMITED`

### 6.3 场景 B：列表查询高并发

目标：

- 验证高频列表查询不拖垮数据库

Locust 用户类型：

- `OperatorReadUser`
- `AdminReadUser`

脚本职责：

- 持续分页、筛选、搜索
- 覆盖列表页核心入口

关键接口：

- `GET /api/bidding/products`
- `GET /api/fulfillment/orders`
- `GET /admin/api/users`
- `GET /api/tasks`

通过标准：

- 不超时
- 不出现明显热表扫描雪崩

### 6.4 场景 C：长任务创建削峰

目标：

- 验证任务化接口削峰与排队

Locust 用户类型：

- `SyncTriggerUser`
- `RecalcTriggerUser`
- `BidBatchTriggerUser`

脚本职责：

- 并发创建任务
- 轮询任务状态
- 统计 duplicate / existing-task 返回

关键接口：

- `POST /api/stores/{id}/sync`
- `POST /api/bidding/stores/{id}/start`
- `POST /api/finance/snapshots/recalculate`
- `GET /api/tasks/{id}`

通过标准：

- 无重复任务失控
- 返回 `task_id` 稳定
- 队列积压可观测

### 6.5 场景 D：高危写操作冲突

目标：

- 验证幂等、乐观锁与审计写入

Locust 用户类型：

- `AdminWriteUser`
- `FulfillmentWriteUser`
- `BiddingWriteUser`

脚本职责：

- 多用户并发改同一对象
- 重复提交同一高危请求
- 校验冲突返回与审计写入

关键接口：

- `POST /admin/api/users/{id}/feature-flags`
- `POST /api/fulfillment/pos`
- `POST /api/fulfillment/pos/{id}/tracking`
- `POST /api/bidding/products/{id}/floor`

通过标准：

- 无重复 PO
- 无重复补号成功
- 返回 `IDEMP_VERSION_CONFLICT`
- 审计记录完整

### 6.6 场景 E：外部平台异常隔离

目标：

- 验证外部异常不拖垮主站

脚本职责：

- 触发外部失败场景
- 同时保持主站读请求压力
- 观察任务状态与错误码

模拟异常：

- Takealot 超时 / 401
- CNExpress 超时
- 1688 / Amazon / AI 失败

通过标准：

- 主站读接口继续可用
- 任务进入正确失败状态
- 前端可消费错误码

## 9. 执行批次

### Batch A：容量预估演练

- 先跑登录与列表查询
- 目标是找出最早瓶颈点

### Batch B：单域压测

- 分别跑 Store / Fulfillment / Bidding / Finance
- 目标是验证单域局部稳定性

### Batch C：P0 全链路压测

- 按真实依赖链组合场景 A~D
- 目标是验证 `P0` 主链

### Batch D：上线前复验

- 结合 Runbook 做彩排
- 做小规模压力复验 + smoke

## 10. 上线前复验清单

- 登录与 `/api/auth/me` 稳定
- Dashboard 汇总读稳定
- 店铺同步创建任务正常
- `PO` 创建与补号稳定
- 高危写接口有审计
- 外部失败不拖垮主站
- 关键指标在门槛内

## 11. 关键冻结点

- 压测工具首选 `Locust`
- 脚本结构按“公共能力 + 场景文件”组织
- 首轮先覆盖 `P0` 场景 A~D
- 外部失败模拟属于必测项，不是可选项
