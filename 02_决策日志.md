# ERP 重构决策日志

## 2026-04-21

### D001：采用战时全量版 PRD 策略

- 决策：重写 `PRD v2.1（战时全量版）`
- 原因：项目必须在 `2026-05-01` 前上线，不能继续沿用理想态节奏
- 影响：文档需要按战时优先级重新组织，而不是沿用原 Phase 顺序

### D002：自动铺货允许延后几天补齐

- 决策：`Listing / Dropship` 自动铺货闭环允许在 `2026-05-03` ~ `2026-05-05` 完成
- 原因：用户明确允许该模块稍慢几天
- 影响：首发文档需把该模块标记为延期补齐，不得与其他模块混为同一上线门槛

### D003：必须补齐平台级超级管理员

- 决策：新增 `super_admin`
- 原因：现有 `Admin` 角色不足以承载商业化用户管理、账号密码管理、到期控制与功能限制
- 影响：PRD 必须补充账号生命周期、权限矩阵、审计与会话治理

### D004：安全与稳定能力必须前置

- 决策：将权限、限流、幂等、审计、监控、备份、回滚前置到首发 PRD
- 原因：目标是商业化、1000 用户导向、可控可审计
- 影响：原 PRD 的 Phase 7 内容需要前移

### D005：建立持续检查点机制

- 决策：通过线程自动检查点 + 本地控制文档保存上下文和任务进度
- 原因：项目推进节奏快，必须防止上下文丢失
- 影响：每次新一轮工作都必须先读取控制文档，再继续推进

### D006：前端采用商业 SaaS 执行控制台风格（已被 D007 修正）

- 决策：前端采用左侧深色导航、右侧浅色内容区、高密度表格、状态徽标、详情抽屉、任务进度中心的商业 SaaS 控制台风格
- 原因：系统面向高频运营、仓库与管理员，不适合营销化、低密度或强动画风格
- 影响：该方向已被 D007 升级为极简 Apple-like 任务型工作台；仍保留“状态真相、下一步动作、高风险确认、长任务可追踪”的原则

### D007：前端视觉方向升级为极简 Apple-like 任务型工作台

- 决策：前端从传统高密度后台风格升级为极简、留白充足、卡片式、高信噪比、任务导向的 Apple-like 商业工具风格
- 原因：用户明确希望减少传统 ERP 拥挤感，用更清爽、更傻瓜式、更适配平板和移动端的交互承载复杂业务
- 影响：默认表格字段减少到 5 到 6 个核心字段，核心模块采用卡片、抽屉、看板、细进度条、视觉化对账、轻量反馈和智能预填

### D008：PRD v2.1 五个上线附录冻结

- 决策：新增并冻结权限矩阵、状态机总表、错误码表、外部接口重试策略、上线门槛与回滚条件五个附录
- 原因：`2026-05-01` 上线需要明确权限边界、业务状态、失败语义、外部依赖容错和事故回滚策略
- 影响：后续 API、前端和 worker 设计必须遵守五个附录，不得绕过权限、状态机、错误码、重试与回滚约束

### D009：API 写接口统一规则冻结

- 决策：新增并冻结 `附录F_API写接口统一规则.md`
- 原因：写接口是最容易失控的部分，必须统一请求头、幂等、审计、任务化和分域规则
- 影响：后续所有写接口设计都必须声明权限、状态前置条件、幂等键、错误码和返回语义，禁止页面导向的临时写法

### D010：任务系统字段与状态全集冻结

- 决策：新增并冻结 `附录G_任务系统字段与状态全集.md`
- 原因：任务系统是全局运行骨架，必须明确字段全集、状态全集、事件全集、租约、重试和死信规则
- 影响：后续所有长任务、worker、watcher、进度展示和取消逻辑必须遵守附录 G，不得私自新增不可追踪状态

### D011：财务快照和流水重算规则冻结

- 决策：新增并冻结 `附录H_财务快照和流水重算规则.md`
- 原因：财务是最不能“差不多”的域，必须保证流水不可变、快照可重算、调整有痕迹
- 影响：后续财务实现不得覆盖流水，不得静默改历史快照，所有重算必须任务化并版本化

### D012：审计字段与高危动作清单冻结

- 决策：新增并冻结 `附录I_审计字段与高危动作清单.md`
- 原因：商业化系统必须能追责、能回溯、能发现异常操作，尤其是账号、权限、竞价、PO、财务和任务恢复
- 影响：后续所有高危动作必须写审计，必须有 request_id、actor、target、before、after、reason、result，不允许审计缺失的静默成功

### D013：PRD v2.1 核心冻结，进入实施准备阶段

### D014：实施文档包首版展开完成

- 影响：后续实现、联调、测试、彩排和上线一律以 04~08 为统一输入，新增细化优先更新实施文档和控制文档，而不是继续堆正文描述

### D017：API 契约主版本统一为详细展开文档

- 原因：接口文档一旦出现双轨草稿，很快就会在角色、错误码和前置条件上产生不一致
- 影响：后续接口拆解、前端联调、测试用例、压测入口和上线验收全部以该文件为准

### D018：建立项目级半自动检查点系统

- 决策：在 `.claude/settings.json` 中启用 `SessionStart`、`UserPromptSubmit`、`PreCompact`、`Stop` 钩子，并新增 `09_检查点模板.md` 作为标准检查点模板
- 原因：项目推进快，完全手动维护控制文档容易漏；但直接自动改写控制文档风险过高
- 影响：后续该项目会在开工、提交新任务、压缩上下文和结束工作时自动提醒维护上下文，并使用模板统一检查点口径

### D019：P0 接口实现按依赖链批次推进

- 决策：将 `P0` 接口实现拆分为 Batch 1~4，按“身份与控制面 → 任务骨架与店铺同步 → 履约主链 → 竞价主链”的顺序推进
- 原因：按模块平铺会掩盖真实依赖关系，导致联调顺序和上线顺序错位
- 影响：后续接口开发、联调、验收和排期应以批次顺序为准，而不是按模块各自并行冲刺

### D020：数据库迁移按“基础设施 → 主链 → 扩展”批次推进

- 决策：将数据库迁移拆分为 Batch 1~6，按“身份权限与任务骨架 → 店铺与连接器 → 履约 → 竞价 → 财务 → P1/P2 扩展”的顺序推进
- 原因：如果仅按业务域平铺迁移，接口开发顺序、联调顺序和上线顺序会错位，回滚边界也不清晰
- 影响：后续 DDL、索引、分区和迁移脚本应以该批次顺序落地，并与 `10_P0接口实现批次拆解.md` 保持一致

### D021：实现拆解进入第二阶段，优先补齐 P1 与表级落地清单

- 决策：在完成 `P0` 接口批次拆解与数据库迁移批次拆解后，下一阶段优先补齐 `P1` 接口实现批次，并把数据库批次继续细化为表级 DDL、索引、回滚与验收清单
- 原因：批次框架已形成，如果不继续下沉到 `P1` 和可执行落地清单，团队仍会停留在“方向明确但无法开工”的状态
- 影响：后续工作优先级调整为 `P1` 接口 → Batch 1~4 DDL 清单 → 页面实施清单 → 压测脚本 → 上线责任矩阵

### D022：前端页面实施按“页面壳 + 任务流 + 通用交互件”推进

- 决策：将前端实施清单按全局框架、业务任务流和通用交互件拆解，而不是单纯按页面名平铺
- 原因：直接按页面名拆会重复定义抽屉、弹层和通用组件，后续实现容易发散
- 影响：后续前端页面开发、组件拆分和联调依赖应以 `12_前端页面实施清单.md` 为主版本

### D023：Fulfillment 前端主交互冻结为双 Tab + 沉浸式打单 Dialog

- 决策：`Fulfillment` 页面采用 `订单池 Tab` 与 `PO 工作台 Tab` 双阶段结构，`PO` 打单使用居中 80% 宽度毛玻璃 Dialog，而不是右侧抽屉
- 原因：该域的核心效率目标不是浏览信息，而是“挑货合单”和“沉浸打单”，需要明确切断干扰
- 影响：后续 Fulfillment 前端实现必须按该交互冻结点推进，跨仓合单前端需置灰并提示，后端继续保留强校验

### D024：压测准备按“环境 → 造数 → Locust 场景 → 观测 → 复验”闭环推进

- 决策：将压测准备拆为环境、造数、Locust 场景脚本、观测项、失败记录模板、执行批次与上线前复验清单
- 原因：如果只保留场景描述而没有脚本结构、造数职责和观测规则，压测工作无法真正开工
- 影响：后续压测实施、环境准备、失败复盘和上线前复验应以 `13_压测脚本与造数方案.md` 为唯一主版本

### D025：压测工具首选 Locust

- 决策：首轮压测默认以 `Locust` 作为主工具，而不是维持工具无关状态
- 原因：ERP 压测需要登录、长任务轮询、高危写冲突和外部异常模拟，使用 Python 更利于表达复杂业务流程
- 影响：后续压测脚本目录、用户类型、场景编排和造数辅助脚本默认围绕 `Locust` 组织

### D026：前端与压测下沉文档进入唯一主版本

- 决策：将 `12_前端页面实施清单.md` 与 `13_压测脚本与造数方案.md` 视为 `06`、`07` 的下沉执行主版本
- 原因：如果仍把 `06`、`07` 当作唯一输入，前端实施和压测准备会缺少组件级、脚本级与职责级口径
- 影响：后续页面开发、组件拆分、Locust 脚本组织、造数协作和压测复验优先以 `12`、`13` 为执行输入

### D027：上线准备按“责任矩阵 → 彩排剧本 → 开关表 → 降级/回滚口令”闭环推进

- 决策：将上线准备继续拆为责任矩阵、彩排剧本、开关表、Smoke 验收表、降级决策矩阵、回滚口令与值守沟通节奏
- 原因：如果只保留 Runbook 主线而不细化责任、口令和开关表，上线当天仍会因为等待确认、动作冲突或信息不对称而返工
- 影响：后续上线演练、正式发布、降级与回滚应以 `14_上线彩排与责任矩阵.md` 为唯一执行主版本

### D028：上线当天开关控制采用“super_admin 后台预置 + 发布负责人统一口头确认”

- 决策：应急开关预置在 `super_admin` 后台，但实际执行仍必须由发布负责人统一口头确认，各域负责人或操作人不能自行跳过确认
- 原因：完全中心化手工切换会慢，完全分权又容易在高压场景下失控，需要兼顾速度与控制
- 影响：后续开关表、彩排剧本、降级矩阵和回滚流程都必须保留“确认人 / 执行人 / 执行时间”字段

### D029：Batch 1~4 数据库迁移继续下沉到表级 DDL 清单

- 决策：将 Batch 1~4 继续细化为字段级、接近 SQL 粒度的表级 DDL 落地清单
- 原因：如果只停留在“批次 + 表名”层级，开发在真正写 migration 时仍会重复讨论字段、约束、索引时机和回滚边界
- 影响：后续 SQL 迁移脚本、DDL review 和上线前数据库验收应以 `15_Batch1-4表级DDL落地清单.md` 为执行基准

### D030：P1 接口实现按“Selection/Extension → Warehouse → Finance”批次推进

- 决策：将 `P1` 接口拆分为 Batch 5~7，按“Selection / Extension → Warehouse → Finance”的顺序推进
- 原因：`P1` 的重点是受限可用而不是阻断主链，如果按模块平铺推进，容易把风险最高的 Finance 提前做重
- 影响：后续 `P1` 接口开发、联调和灰度上线应按该顺序推进，不与 `P0` 主链抢节奏

### D031：Finance 在 P1 阶段采用“读侧优先 + 白名单重算 + 非默认全面 adjustment”

- 决策：`Finance` 在 `P1` 阶段优先开放读侧，`recalculate` 只对白名单或受限范围开放，`adjustments` 不做默认全面开放
- 原因：财务域风险最高，若过早全面开放写能力，会直接增加数据失控与审计风险
- 影响：后续前端、后端、压测和上线验收都必须按该受限边界设计，不得把 `Finance` 当作完整开放域

### D032：前端页面开发顺序按 Wave 1~4 冻结

- 决策：将前端页面开发顺序冻结为 Wave 1 全局框架与控制面、Wave 2 Fulfillment 主链、Wave 3 Store/Sync 与 Bidding、Wave 4 Finance/Warehouse/Selection/Extension
- 原因：如果不固定开发波次，前端很容易在通用件未稳定时并行开发多个业务页面，导致返工
- 影响：后续页面开发排期、联调顺序与组件复用应按该 Wave 顺序推进

### D033：Admin 高危交互采用“统一骨架 + 动作差异化风险说明”

- 决策：Admin 域高危动作统一使用确认骨架，但根据动作类型展示不同风险说明与按钮文案
- 原因：完全分散会导致交互风格漂移，完全统一文案又会让风险语义失真，需要兼顾一致性与可理解性
- 影响：后续 Admin 前端实现、设计稿和验收都必须按该规则执行，不得私自新增非标准确认框

### D034：压测执行采用“目录/职责/批次/三档结果”统一冻结口径

- 决策：将压测执行继续冻结为目录结构、Locust 用户类型、造数职责、执行批次、三档通过标准、失败模板与上线前复验口径
- 原因：如果压测只停留在准备方案层，真正执行时仍会在目录、职责、报告口径与是否通过上反复争论
- 影响：后续压测脚本编写、执行、复盘和上线前复验应以 `18_压测执行口径冻结.md` 为唯一执行主版本

### D035：压测结果采用“可上线 / 理想通过 / 阻断上线”三档判定

- 决策：压测结果不采用简单二元通过，而采用“可上线 / 理想通过 / 阻断上线”三档判定
- 原因：ERP 首发阶段需要在时间压力下做受控决策，既不能因为不完美就无限延期，也不能因为勉强通过就掩盖重大风险
- 影响：后续压测报告、上线评审和发布窗口决策必须按三档表达，不能只写 pass/fail

### D036：进入“整体一致性复查与收口”阶段

- 决策：在完成接口、DDL、页面、压测与上线细化文档后，下一阶段以一致性复查与收口为主，而不是继续横向扩展新文档
- 原因：当前风险已从“缺文档”转为“文档之间潜在漂移”，如果不先收口，后续实现阶段会在细节冲突处返工
- 影响：后续工作应优先依据 `19_一致性复查与收口清单.md` 逐项检查并收敛口径，再进入正式实现准备

- 决策：浏览器扩展默认交互改为页面右下角登录卡：用户直接在扩展内输入 `ERP 地址 + 账号 + 密码` 登录；后台通过 `POST /api/extension/login` 静默换取 `Bearer token` 并保存在扩展本地存储
- 原因：手工复制 token 的操作成本过高，不适合商业化；用户真正关心的是“能否登录、能否选店铺、能否设置保护价”，而不是 token 概念本身
- 影响：当前扩展骨架已改为三态：未连接登录卡、已连接商品护栏卡、店铺选择联动；Popup 继续作为补充配置入口，不再是主流程入口

### D064：商品事实补全采用“三段式来源优先级”

- 决策：`profit-preview` 的商品事实补全采用三段式优先级：先读 `library_products`，其次尝试当前店铺 listing payload 中的 `weight_grams/length_cm/width_cm/height_cm`，最后在配置了 `seller-api` 凭证时调用 `GET /1/catalogue/mpv/PLID<PLID>`
- 原因：共享事实库必须是首选，避免重复外呼；listing payload 是当前系统最稳的低成本事实来源；`seller-api` 可作为更完整的全量商品事实补全来源，但必须在有后台服务凭证时才启用
- 影响：当前闭环即便未配置 `seller-api` 也能通过 listing payload 回填出基础重量尺寸；一旦配置 `XH_TAKEALOT_CATALOG_EMAIL/PASSWORD` 或 `XH_TAKEALOT_CATALOG_API_KEY`，即可进一步补全更完整的包装字段

### D050：首发信息架构不按竞品菜单数对齐，而按主线程闭环收口

- 决策：首发版本的导航与页面边界以 24_首发信息架构与页面改造蓝图.md 为统一主版本，不按竞品的菜单数量和拆页方式进行 1:1 对齐。
- 原因：竞品导航层级过多，商品、库存、上架、竞价、分析被拆得过碎，会提高学习成本并干扰首发主线程。
- 影响：后续页面设计和前端改造统一围绕 工作台 / 店铺管理 / 上架中心 / 自动竞价 / 上架记录 / 订单中心 / 管理员控制面 展开，其余模块延后或并入二级结构。

### D066：店铺管理页定位为商业操作页，不做技术排障页

- 决策：`/stores` 首发页只承载店铺绑定、凭证更新、同步、对账/校准、上架记录视角与 SKU 保护底价策略；任务队列、内部字段、脱敏 Key 和排障细节不进入普通店铺页。
- 原因：用户已确认前端必须是黑白、Apple 风格、极简、高端、商业化；店铺管理应帮助用户完成主线程闭环，而不是暴露运维噪音。
- 影响：`/stores?tab=strategies` 作为自动竞价入口之一继续保留；单条策略用右侧 Sheet 编辑，批量底价用轻量导入，不再恢复旧页面大而杂的信息堆。

### D067：前端视觉方向改为竞品式高密企业工具，而非营销式大卡片

- 决策：首发前端对标用户视频里的辉光 ERP：细白侧栏、顶部状态栏、页面内横向页签、白底细边框卡片、高密表格和轻量操作条；禁止继续使用黑色大侧栏、大标题 Hero、大面积留白和营销式卡片堆叠。
- 原因：用户明确指出上一版“瞎搞”，竞品真实形态是运营后台，不是品牌官网；商业化质感来自信息密度、控件克制和任务效率，而不是视觉装饰。
- 影响：`/dashboard`、`/stores`、`/stores?tab=strategies` 已按此方向重做；后续 `/orders`、`/listing`、`/admin`、`/tasks` 也必须沿用同一高密白底系统。

### D068：自动竞价必须独立成页，不再挂在店铺管理 Tab 下

- 决策：`自动竞价` 首发作为独立一级页面 `/bidding`，承载 SKU 保护底价、策略状态、单条策略 Sheet 和批量导入；`店铺管理` 不再展示 SKU 策略 Tab。
- 原因：自动竞价是“管价”主线程，不是店铺资料的附属设置；混在店铺页会破坏 IA，也会让用户误以为策略只是店铺配置项。
- 影响：导航已切到 `/bidding`；`/stores` 仅保留店铺接入、凭证、同步和店铺视角上架记录。

### D069：竞品只能参考信息密度，不能复制视觉图形

- 决策：首页不再使用竞品味明显的热力图/矩形商品块，改用原创经营指标、经营脉冲、处理队列和状态表。
- 原因：用户明确反馈“抄袭味道太重”；设计目标是黑白极简、高端商业工具，而不是复刻竞品屏幕。
- 影响：后续页面只能借鉴“高密、克制、低噪声”的原则，不复制竞品具体图形、布局组合和命名方式。

### D070：暂停当前前端视觉探索，改为先输出逻辑交接给 Gemini

- 决策：当前不再继续由 Codex 直接推进前端视觉设计，先把 IA、业务流程、页面职责、接口和权限逻辑交接给 Gemini 做原创视觉方案。
- 原因：现有多轮设计仍未达到用户预期，继续直接改页面会浪费时间；先分离“产品逻辑”和“视觉设计”更稳。
- 影响：后续 Codex 优先推进非视觉工程事项；前端视觉以 `25_Gemini前端设计逻辑交接稿.md` 为输入，等 Gemini 方案确定后再实现。

### D071：正式异常值班以 `21_异常处理与人工介入手册.md` 为唯一主版本

- 决策：将 `21_异常处理与人工介入手册.md` 冻结为首发值班、任务恢复、人工介入权限和客户沟通的唯一内部主版本；`docs/commercial_delivery/incident_response_playbook.md` 仅作为客户交付子集。
- 原因：D039 / D040 已冻结结构和权限级别，但如果没有正式手册，值班仍会在错误码表、任务附录、上线 Runbook 和客户交付文档之间来回跳。
- 影响：后续异常培训、任务恢复、履约错绑处理、店铺同步鉴权失败和财务受控动作，必须优先引用 `21_异常处理与人工介入手册.md`。
