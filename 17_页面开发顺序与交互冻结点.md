# 页面开发顺序与交互冻结点

## 1. 目标

本文件继续下沉 `12_前端页面实施清单.md`，把前端实施从“有页面清单”推进到“有开发顺序、组件依赖顺序和交互冻结点”。

目标：

- 防止前端开发顺序反复调整
- 防止通用组件边界反复改口
- 防止高危交互在实现中途漂移

---

## 2. 页面开发总原则

- 先全局壳，再任务主链，再受限域
- 先通用件，再业务页面
- 先冻结高危动作，再写交互细节
- 先保证任务闭环，再补低优先级视觉细节

---

## 3. 页面开发波次

### Wave 1：全局框架与控制面

范围：

- `AppShell`
- `SidebarNav`
- `TopStatusBar`
- `GlobalToast`
- `Task Drawer`
- `Audit Detail Drawer`
- `Dashboard`
- `Admin`

目标：

- 先把全局壳与控制面打稳
- 让后续所有模块共享统一骨架和高危确认能力

### Wave 2：Fulfillment 主链

范围：

- `Fulfillment / 订单池 Tab`
- `Fulfillment / PO 工作台 Tab`
- `PoTicketingDialog`
- `异常标记弹层`

目标：

- 优先交付最重要的业务执行体验
- 固化双 Tab 与沉浸式打单流程

### Wave 3：Store / Sync 与 Bidding

范围：

- `Store 列表 / 详情`
- `凭证更新抽屉`
- `同步任务抽屉`
- `Bidding 工作台`
- `单品详情抽屉`
- `全店竞价确认弹层`

目标：

- 承接店铺与竞价执行链
- 在已有 Task / Admin / Fulfillment 骨架上稳定复用组件

### Wave 4：Finance / Warehouse / Selection / Extension

范围：

- `Finance`
- `Warehouse`
- `Selection`
- `Extension`

目标：

- 作为 `P1` 受限能力补齐
- 不反向阻断前 3 个波次

## 5. 高危交互冻结表

### 5.1 统一原则

- 所有高危动作采用统一确认骨架
- 但不同动作必须显示不同风险说明
- 不允许页面各自发明确认文案结构

### 5.2 冻结格式

每个高危确认弹层固定包含：

- 操作对象
- 影响范围
- 是否可恢复
- 风险说明
- 原因输入框
- 确认按钮文案

### 5.3 Admin 域

| 动作 | 风险说明 | 确认按钮 |
|---|---|---|
| 重置密码 | 用户将被迫重新建立登录凭据，旧会话可能失效 | `确认重置密码` |
| 禁用用户 | 用户将立即失去访问能力，相关写操作可能被阻断 | `确认禁用用户` |
| 设置到期时间 | 到期后核心写能力可能被关闭 | `确认设置到期` |
| 修改功能权限 | 可能改变用户可见菜单与写权限边界 | `确认更新权限` |

### 5.4 Fulfillment 域

| 动作 | 风险说明 | 确认按钮 |
|---|---|---|
| 合并为 PO | 订单项会被绑定到同一 `PO`，后续需按该单据继续补号与履约 | `确认合并为 PO` |
| 取消 PO | 该 `PO` 将退出主链，需要人工重新分配后续动作 | `确认取消 PO` |
| 标记异常 | 对象将退出正常执行流，进入人工处理路径 | `确认标记异常` |

### 5.5 Bidding 域

| 动作 | 风险说明 | 确认按钮 |
|---|---|---|
| 启动全店竞价 | 该店铺下符合条件的商品将进入自动竞价流程 | `确认启动竞价` |
| 停止全店竞价 | 该店铺自动竞价执行将立即停止 | `确认停止竞价` |
| 导入底价 | 将批量影响 SKU 安全边界与后续竞价判断 | `确认导入底价` |

### 5.6 Finance 域

| 动作 | 风险说明 | 确认按钮 |
|---|---|---|
| 触发重算 | 将创建重算任务，可能影响后续展示版本 | `确认发起重算` |
| 财务调整 | 会生成受审计 adjustment，影响财务口径 | `确认执行调整` |
| 冻结快照 | 当前版本将被视为正式口径 | `确认冻结快照` |

## 6. 长任务反馈冻结表

### 6.1 统一展示元素

所有长任务必须展示：

- `status`
- `stage`
- `progress`
- 最近事件
- 错误码 / 错误摘要
- 可取消 / 可重试入口（如允许）

### 6.2 统一入口

- 顶部状态条任务入口
- `Task Drawer`
- 业务页面内局部任务入口

### 6.3 不允许的做法

- 只有 spinner
- 无错误码
- 无 task_id 回链
- 页面成功但后台实际失败

## 7. 页面级冻结点

### 7.1 Dashboard

- 必须只消费汇总读接口
- 不允许页面自己拼接业务真相

### 7.2 Admin

- 高危弹层统一骨架，不同动作不同风险说明
- 详情、最近审计优先抽屉，不跳页

### 7.3 Fulfillment

- 双 Tab 结构冻结
- `PO` 打单必须用居中 `Dialog`
- 禁止跨仓合单必须按钮置灰 + 黑白 Toast

### 7.4 Store

- 凭证更新走抽屉，不跳页
- 保存后返回任务入口，不等待同步完成

### 7.5 Bidding

- Slider 改值不自动保存
- 全店竞价启停必须确认

### 7.6 Finance

- 读侧优先
- 高危动作确认必须保留口径说明

---

## 8. 页面验收门槛

### Wave 1 门槛

- 全局壳稳定
- Dashboard / Admin 可联调
- 高危确认骨架统一

### Wave 2 门槛

- Fulfillment 双 Tab 可跑通
- `PO` 打单 Dialog 可连续录入
- 异常流可回链审计

### Wave 3 门槛

- 店铺同步流和竞价主交互可跑通
- 任务回链稳定

### Wave 4 门槛

- `P1` 页面边界明确
- 不展示误导性完整能力

---

## 9. 关键冻结点总结

- 页面开发顺序按 Wave 1 → 4 推进
- 通用组件先于业务复用件
- 高危交互采用统一骨架 + 动作差异化风险说明
- 长任务反馈必须统一，不允许纯 spinner
- `Fulfillment` 双 Tab 与沉浸式 `Dialog` 不再改口
