# ProfitLens / 小黑 ERP
# 系统重构核心产品需求文档（PRD v2.1 战时全量版）

**文档版本**：v2.1  
**日期**：2026-04-21  
**文档定位**：面向 `2026-05-01` 上线窗口的战时全量版 PRD  
**适用对象**：产品、架构、前端、后端、测试、运维、外包协作方  
**当前策略**：除 `Listing / Dropship` 自动铺货闭环外，其余核心域以 `2026-05-01` 为上线目标；自动铺货允许在 `2026-05-03` ~ `2026-05-05` 补齐

---

# 0. 执行摘要

ProfitLens / 小黑 ERP 的重构目标，不是再造一个后台，而是在极短时间内落地一套面向 Takealot 卖家的执行型 ERP。

本版 PRD 采用战时策略：

1. 保留全域能力，不再删掉核心模块  
2. 对能力进行分级：`生产可用`、`受限可用`、`延期补齐`  
3. 把管理员、权限、安全、幂等、审计、监控前置  
4. 用状态机和任务系统收敛所有高风险长流程  
5. 明确 `2026-05-01` 上线门槛，避免“功能看起来都有，实际不可控”

一句话定位：

> ProfitLens 是一套面向 Takealot 卖家的垂直运营 ERP，通过店铺同步、商品情报、自动铺货、自动竞价、履约执行、财务快照与平台级管理员控制台，将卖家的日常经营动作转化为可执行、可审计、可扩展的交易系统。

# 1. 上线目标、范围与战时策略

## 1.1 总目标

系统必须在 `2026-05-01` 前具备商业化最小可用能力，并满足以下条件：

- 可登录、可控权、可限制功能
- 可管理用户账号、密码、到期时间和可用功能
- 可接入和同步 Takealot 店铺
- 可执行履约主链并形成 PO 工作台
- 可执行受护栏保护的自动竞价
- 可沉淀基础财务快照与费用结果
- 可提供基础仓库与物流协同能力
- 可在高压场景下保持可观测、可限流、可回滚

## 1.2 上线范围分级

### `P0`：`2026-05-01` 前必须生产可用

- `Auth / Admin / Subscription`
- `Store Management`
- `AutoBid`
- `Fulfillment / PO Workbench`
- 安全与运行底座

### `P1`：`2026-05-01` 前受限可用

- `Product Intelligence / Selection`
- `Finance`
- `Warehouse / CNExpress`
- `Extension`

### `P2`：`2026-05-03` ~ `2026-05-05` 补齐闭环

- `Listing / Dropship`

## 1.3 非目标

以下能力不属于本轮重构目标：

- 微服务拆分
- 多平台通用 ERP
- 原生移动端
- 复杂多租户组织树
- 通用 BI 平台

# 2. 架构红线

1. 前端不得推导业务真相  
2. 后端必须返回 canonical `status / stage / ui_meta`  
3. 高风险写操作必须带权限校验、幂等保护、审计记录  
4. 所有长耗时动作必须任务化与异步化  
5. 所有外部平台必须经过 adapter 与错误映射层  
6. 自动竞价必须受 floor / ceiling 护栏约束  
7. PO 写路径必须集中收口，不允许多入口竞争写入  
8. 财务流水必须可追溯，利润快照必须可重算  
9. 管理员后台必须具备独立安全策略与高危操作留痕

# 3. 平台角色与权限模型

## 3.1 角色定义

### `super_admin`

平台最高权限角色，负责：

- 创建、禁用、冻结用户
- 重置密码、强制改密、强制下线
- 设置账号到期时间与宽限期
- 配置功能权限与资源限额
- 查看系统健康、审计日志、任务运行状态

### `tenant_admin`

租户侧管理员，负责：

- 管理本租户下普通用户
- 控制本租户内功能启用范围
- 查看本租户的业务运行与异常情况

### `operator`

业务执行角色，负责：

- 选品
- 铺货
- 商品管理
- 履约跟进
- 竞价观察

### `warehouse`

仓库执行角色，负责：

- 到仓
- 扫描
- 打包
- 贴标
- 出库

### `system_worker`

系统任务执行角色，负责：

- AutoBid Runner
- Listing Worker
- Dropship Worker
- Fulfillment Import Worker
- Submission Sync Worker
- Recovery Watcher
- Webhook Processor

## 3.2 权限原则

- 角色权限不只由页面控制，必须由后端接口强制执行
- 高危操作必须记录 `actor`、`target`、`before`、`after`、`reason`
- 已完成或已取消对象默认只读
- `super_admin` 可做平台级干预，但仍需审计

- 用户过期后默认禁止核心写操作

- 用户被禁用后，其未完成任务必须收敛到可回收状态
- 功能被关闭后，历史数据保留只读能力

# 7. 业务域与上线要求

## 7.1 Auth / Admin / Subscription

### 首发目标

- 登录、登出、当前用户
- 平台级超级管理员
- 密码管理
- 到期时间管理
- 功能权限开关
- 会话治理
- 审计日志

### 首发要求

- `/api/auth/me` 为登录态唯一真相
- 管理员高危操作必须审计
- 限流、会话控制、强制登出必须可用

## 7.2 Store Management

### 首发目标

- 店铺接入
- 凭证校验
- 店铺同步
- 健康检查
- Webhook 配置

### 首发要求

- 店铺同步必须任务化
- 店铺同步成功/失败必须回写
- 不允许无幂等地重复导入同一批数据

## 7.3 Product Intelligence / Selection

### 首发目标

- 商品情报同步
- 搜索、筛选、标注
- 候选商品池
- 批量导出

### 首发要求

- 以只读分析和筛选为主
- 不与自动铺货形成硬耦合

## 7.4 Listing / Dropship

### 战时策略

本模块保留在 PRD 正文中，但允许在 `2026-05-03` ~ `2026-05-05` 补齐。

### 能力目标

- 链接直铺
- 关键词采买
- AI 改写
- 类目匹配
- loadsheet 生成
- 提交审核
- submission / review 状态回写

### 强制规则

- Brand 固定为 `Generic`
- 初始售价留空
- 必须有重试、驳回返修、重新提交闭环

## 7.5 AutoBid

### 首发目标

- 在售商品同步
- BuyBox 获取
- SKU 底价导入
- 自动调价
- 调价日志
- 店铺级启停

### 强制规则

- 无人工底价不得竞价
- 理想价 = `BuyBox - 1 ZAR`
- `ceiling = synced_base_price * 1.10`
- 所有执行必须写 `bid_log`

## 7.6 Fulfillment / PO Workbench

### 首发目标

- 官方订单导入
- 订单池
- 同仓合单
- 生成 PO
- 逐单补号
- 物流 / 仓库推进

### 强制规则

- 禁止跨仓合单
- `PO_Manager` 为唯一 PO 写路径 owner
- 已完成 / 已取消对象默认只读

## 7.7 Finance

### 首发目标

- 钱包
- 钱包流水
- 物流费用
- 利润快照

### 强制规则

- 钱包流水必须可追溯
- 利润快照必须可重算
- 首发允许“读强于写”，但不可缺可追溯链

## 7.8 Warehouse / CNExpress

### 首发目标

- 到仓
- 扫描
- 打包
- 贴标
- 出库批次
- 物流轨迹

### 首发要求

- 仓库动作必须留痕
- 物流异常必须可标记与追踪

## 7.9 Extension

### 首发目标

- 最小鉴权
- 快速触发利润计算
- 快速跳转到系统内目标页面

### 首发限制

- 不做复杂自动写入

# 8. 数据真相与一致性原则

- 前端不创造状态
- 所有外部数据先标准化后入核心业务表
- 跨域一致性采用：
  - 本地提交
  - outbox 事件
  - 下游幂等消费
- 财务流水只追加不覆盖
- 所有高风险对象必须定义 canonical truth source

---

# 9. 外部系统 Adapter 策略

所有外部连接器必须具备：

- 原始 payload 留存
- 标准化映射
- 去重键
- 超时重试
- 错误分级
- 死信与隔离

优先级：

- `P0`：Takealot、CNExpress
- `P1`：1688、Amazon、DeepSeek

---

# 10. 安全与稳定基线

系统必须在首发前具备以下能力：

- 登录限流
- API 限流
- RBAC
- 幂等保护
- 审计日志
- 会话管理
- WAF / CDN / 反向代理
- 密钥与凭证隔离
- 错误信息脱敏
- 监控、告警、备份、回滚

关键原则：

> 接口无法被真正隐藏，只能通过权限、限流、隔离、审计与防护来避免被滥用或击穿。

---

# 11. 面向 1000 用户的非功能目标

- 设计目标：支持 `1000` 用户同时在线
- 读流量优先走缓存或汇总视图
- 写流量优先走任务队列削峰
- 外部依赖故障不能拖垮主站
- 管理流量与业务流量隔离
- 关键任务不得重复执行
- 关键调价不得重复提交
- 财务数据不得重复入账

# 12. 前端风格与交互设计

## 12.1 设计定位

前端采用极简、清爽、低学习成本的现代商业工具风格。

它不是传统 ERP 的拥挤表格，也不是营销站，而是一个让用户“少思考、快操作、少犯错”的任务型工作台。

设计关键词：

- 极简主义
- 呼吸感
- 高信噪比
- 傻瓜式流程
- 低视觉疲劳
- 状态清晰
- 高风险动作可控

## 12.2 视觉语言

- 主背景使用浅灰色或纯白
- 主要内容区使用圆角白色卡片
- 主操作色采用 Apple 风格蓝色，接近 San Francisco Blue
- 状态标记使用低饱和度语义色
- 通过留白、字号、字重和层级区分信息主次
- 图标使用细线条 Outline Icons
- 字体优先使用系统原生字体，如 San Francisco 或 Segoe UI

## 12.3 卡片式任务布局

核心任务必须尽量封装在卡片中。

适合卡片化的对象：

- 自动竞价设置
- 店铺健康
- 待处理订单
- PO 摘要
- 任务进度
- 账号到期提醒
- 系统风险提示

卡片内必须有清晰标题、关键指标、状态和主操作。

## 12.4 任务导向导航

- 一级导航只保留最高频模块
- 二级功能通过抽屉式菜单、页面内 Tab 或局部操作展开
- 用户应尽量停留在当前任务流中，不被多层页面跳转打断
- 导航必须根据功能授权动态展示或禁用，但后端仍必须进行权限校验

首发一级导航建议：

- Dashboard
- Selection
- Listing
- Bidding
- Fulfillment
- Finance
- Admin

## 12.5 交互逻辑

- 系统应根据历史数据预填常用字段
- 创建上架任务时，应根据关键词自动推荐类目
- 合单时，应根据目的仓自动筛选可合单订单
- 保存成功时，按钮可短暂转为绿色对勾
- 成功反馈使用轻量 toast，不使用巨大遮罩
- 高风险失败必须在页面内明确展示
- 长任务开始后必须给出任务入口

理想操作路径：

```text
系统推荐
  → 用户确认
  → 系统执行
  → 用户查看结果
```

## 12.6 数据呈现

- 默认只展示最核心的 5 到 6 个字段
- 高级字段默认隐藏
- 用户可切换高级视图
- 用户可自定义列显示
- 系统必须记住用户的列偏好
- 表格必须服务端分页、筛选、排序

履约中心必须进行视觉化对账：

- 商品图片与物流单号并列显示
- 商品图片与订单信息同屏展示
- PO 内逐单补号时必须能看到商品图
- 仓库操作时优先展示图片、SKU、数量和目的仓

## 12.7 任务进度感知

耗时任务不得只显示 spinner。

必须展示：

- 顶部细进度条
- 当前阶段
- 已完成数量
- 失败数量
- 最近事件
- 可取消 / 可重试入口

适用场景：

- 全站采集
- 店铺全量同步
- 大规模调价
- 批量铺货
- submission / review 同步

## 12.8 关键模块设计亮点

### AutoBid 自动竞价工作台

- 用简洁折线图展示 BuyBox 波动
- 用区间带展示 floor / ceiling 安全边界
- 当前价格、BuyBox、底价、上限必须同屏可见
- 底价调整可使用 Slider，但必须允许精确输入
- Slider 调整不得直接提交，必须点击确认保存
- 调价日志以时间线形式展示

### Listing / Dropship 铺货流水线

- 采用看板模式表达流程
- 支持批量处理
- 支持一键确认系统推荐
- 支持查看 AI 改写前后对比
- 支持驳回返修与重新提交
- 拖拽只能改变内部工作流状态，不得绕过后端状态机

### Fulfillment 履约中心

- 订单池按目的仓聚合
- 同仓可合单订单突出显示
- 商品图片、数量、SKU、物流单号并列
- PO 使用右侧详情抽屉
- 异常单必须一眼可识别

### Admin 管理控制台

- 用户列表默认显示账号、角色、状态、到期时间、功能权限摘要
- 重置密码、禁用、修改到期时间等高危动作进入抽屉
- 权限配置使用清晰开关与套餐模板
- 审计日志以时间线展示

## 12.9 性能与并发约束

- 列表必须分页
- 搜索必须防抖
- Dashboard 必须读汇总接口
- 前端不得高频轮询热接口
- 长任务进度必须退避轮询或使用事件流
- 同一页面不得重复触发相同写请求
- 动画必须克制，不能影响低性能设备操作
