# 数据库迁移批次拆解

## 1. 目标

本文件把 `05_Schema_Blueprint.md` 中的表结构蓝图，拆成可执行的数据库迁移批次、DDL 落地顺序、索引追加顺序与回滚策略。

目标：

- 让表结构落地顺序与 `P0` 接口实现批次保持一致
- 让迁移可控、可回滚、可逐批验收
- 避免一口气建全域大表导致上线窗口失控

---

## 2. 为什么采用方案 B

采用“基础设施 → 主链 → 扩展”的迁移策略，而不是单纯按业务域平铺。

原因：

1. 没有身份、权限、审计、任务骨架，后续接口即使开发完成也无法稳定运行。
2. `Store / Connector` 是外部同步与长任务的入口，必须先于履约和竞价。
3. `Fulfillment` 是上线主链，应优先于 `AutoBid`。
4. `Finance` 需要强一致和 append-only 约束，适合在主链稳定后按独立批次落地。
5. `P1 / P2` 扩展域不应拖慢 `2026-05-01` 主链上线。

---

## 3. 迁移批次总览

| 批次 | 名称 | 目标 | 对应上游 |
|---|---|---|---|
| Batch 1 | 身份、权限、审计、任务骨架 | 打通账号控制面、审计、任务基础设施 | P0 Batch 1 |
| Batch 2 | 店铺、凭证、连接器入口 | 打通店铺、凭证、Webhook、同步入口 | P0 Batch 2 |
| Batch 3 | 履约主链 | 打通订单、PO、Shipment、异常主链 | P0 Batch 3 |
| Batch 4 | 竞价主链 | 打通竞价商品、底价、策略、运行记录与日志 | P0 Batch 4 |
| Batch 5 | 财务与重算 | 打通钱包、快照、汇率、adjustment | P1 / 上线后增强 |
| Batch 6 | P1 / P2 扩展域 | 打通 Selection、Warehouse 扩展、Listing / Dropship | P1 / P2 |

---

## 4. Batch 1：身份、权限、审计、任务骨架

### 4.1 建表范围

- `tenants`
- `users`
- `user_passwords`
- `auth_sessions`
- `user_feature_flags`
- `user_devices`
- `activation_codes`
- `user_subscriptions`
- `licenses`
- `audit_logs`
- `system_health_snapshots`
- `task_definitions`
- `task_runs`
- `task_events`
- `task_leases`
- `task_dead_letters`
- `notifications`

### 4.2 DDL 顺序

1. `tenants`
2. `users` / `user_passwords` / `auth_sessions`
3. `user_feature_flags` / `user_devices` / `activation_codes` / `user_subscriptions`
4. `licenses`
5. `audit_logs` / `system_health_snapshots`
6. `task_definitions`
7. `task_runs`
8. `task_events` / `task_leases` / `task_dead_letters`
9. `notifications`

### 4.3 约束与唯一键

优先在建表时落：

- `users.username`
- `auth_sessions.session_token`
- `user_feature_flags(user_id, feature_key)`
- `licenses.license_key`
- `task_definitions.task_type`
- `task_leases.task_id`

### 4.4 索引顺序

先落最小必需索引：

- `users(tenant_id,status)`
- `users(tenant_id,role)`
- `auth_sessions(user_id,status)`
- `task_runs(status,created_at desc)`
- `task_runs(tenant_id,created_at desc)`
- `task_events(task_id,created_at desc)`
- `audit_logs(tenant_id,created_at desc)`

### 4.5 回滚策略

- 应用回滚优先
- 表保留，暂停引用
- `audit_logs / task_events` 不做物理回滚

### 4.6 验收口径

- `Auth / Admin / Task` 所需最小表已齐
- 高危操作有可落审计表
- 长任务可落 `task_runs + task_events`

## 5. Batch 2：店铺、凭证、连接器入口

### 5.1 建表范围

- `stores`
- `store_credentials`
- `store_feature_policies`
- `takealot_webhook_configs`
- `takealot_webhook_deliveries`
- `connector_inbox`
- `outbox_events`

### 5.2 DDL 顺序

1. `stores`
2. `store_credentials` / `store_feature_policies`
3. `takealot_webhook_configs`
4. `takealot_webhook_deliveries`
5. `connector_inbox`
6. `outbox_events`

### 5.3 约束与唯一键

- `stores(tenant_id,name)`
- `store_credentials.store_id`
- `store_feature_policies.store_id`
- `takealot_webhook_configs.store_id`
- `connector_inbox(provider,external_id,payload_hash)`
- `outbox_events(event_type,aggregate_type,aggregate_id,version)`

### 5.4 索引顺序

- `stores(tenant_id,status)`
- `takealot_webhook_deliveries(store_id,created_at desc)`
- `connector_inbox(provider,created_at desc)`

### 5.5 回滚策略

- 先停 webhook 消费与同步任务
- 保留 inbox / outbox 原始数据，不清空历史

### 5.6 验收口径

- 店铺与凭证读写所需表齐全
- 外部同步原始响应有 inbox 承接
- Webhook 与同步入口具备最小可观测性

## 10. 批次间依赖

```text
Batch 1
  → Batch 2
  → Batch 3
  → Batch 4
  → Batch 5
  → Batch 6
```

说明：

- Batch 1 是所有控制与任务骨架
- Batch 2 承接外部同步入口
- Batch 3 / 4 对应 `P0` 两条业务主链
- Batch 5 / 6 放在主链稳定后推进

---

## 11. 与接口批次的对应关系

| 接口批次 | 对应迁移批次 |
|---|---|
| P0 Batch 1 身份与控制面 | DB Batch 1 |
| P0 Batch 2 任务骨架与店铺同步 | DB Batch 1 + 2 |
| P0 Batch 3 履约主链 | DB Batch 3 |
| P0 Batch 4 竞价主链 | DB Batch 4 |
| P1 财务 / Warehouse / Selection | DB Batch 5 + 6 |

---

## 12. 验收总口径

- 迁移顺序与接口实现顺序一致
- 每批次都可独立回滚或降级
- 高风险 append-only 表不做物理回滚
- `P0` 上线主链所需表必须在 Batch 1~4 完成
- `P1 / P2` 不得反向阻断 `2026-05-01` 上线窗口
