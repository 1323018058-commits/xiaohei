# P0 接口实现批次拆解

## 1. 目标

本文件把 `04_API契约逐接口展开.md` 中 `P0` 范围的接口，按真实依赖链拆成可执行的开发与联调批次，而不是按模块平铺。

拆解目标：

- 让开发顺序符合上线依赖
- 让联调顺序符合前后端与任务系统的真实关系
- 让验收顺序可以直接映射到压测和上线 Runbook

---

## 2. 为什么采用方案 B

采用“按依赖链拆批次”，而不是“按模块拆文档”。

原因：

1. `Auth / Admin` 不先稳住，后面所有模块都没有可信权限边界。
2. `Task / Store Sync` 不先打通，长任务、外部同步、错误回写都没法联调。
3. `Fulfillment / PO` 是 `2026-05-01` 的核心主链，比 `AutoBid` 更早成为上线阻断项。
4. `AutoBid` 依赖身份、店铺、任务、审计和错误码骨架，适合放在最后一批闭环。

---

## 3. 批次总览

| 批次 | 名称 | 目标 | 阻断下游 |
|---|---|---|---:|
| Batch 1 | 身份与控制面 | 打通登录态、权限、功能开关、管理员高危写接口、基础读侧 | 是 |
| Batch 2 | 任务骨架与店铺同步 | 打通任务查询、店铺读写、凭证校验、同步任务创建与状态回写 | 是 |
| Batch 3 | 履约主链 | 打通订单池、PO、补号、异常、Shipment 主链 | 是 |
| Batch 4 | 竞价主链 | 打通底价、策略、全店竞价任务、日志与护栏校验 | 否 |

---

## 4. Batch 1：身份与控制面

### 4.1 范围

接口：

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`
- `GET /admin/api/users`
- `GET /admin/api/users/{id}`
- `POST /admin/api/users`
- `POST /admin/api/users/{id}/reset-password`
- `POST /admin/api/users/{id}/disable`
- `POST /admin/api/users/{id}/enable`
- `POST /admin/api/users/{id}/set-expiry`
- `POST /admin/api/users/{id}/feature-flags`
- `POST /admin/api/users/{id}/force-logout`
- `GET /admin/api/audits`
- `GET /admin/api/system/health`

### 4.2 前置依赖

- 用户、会话、订阅、功能开关基础表
- 权限矩阵与高危动作审计规则
- 基础错误返回结构

### 4.3 开发顺序

1. `login / logout / me`
2. 用户列表与详情读接口
3. 功能开关写接口
4. 重置密码、禁用、启用、强制下线
5. 设置到期时间
6. 审计列表与系统健康读接口

### 4.4 联调顺序

1. 前端先接 `GET /api/auth/me`
2. 再接 Admin 用户列表与详情
3. 再接功能开关弹层
4. 最后联调高危动作确认流

### 4.5 验收口径

- `super_admin / tenant_admin` 边界正确
- 功能关闭时后端拒绝写操作
- 高危动作都有 `Idempotency-Key`、`X-Audit-Reason` 和审计记录
- `/api/auth/me` 能返回角色、功能开关、订阅状态

### 4.6 风险点

- 角色边界写错会污染所有下游接口
- 高危动作没审计会直接影响上线门槛

---

## 5. Batch 2：任务骨架与店铺同步

### 5.1 范围

接口：

- `GET /api/tasks`
- `GET /api/tasks/{id}`
- `GET /api/tasks/{id}/events`
- `POST /api/tasks/{id}/retry`
- `POST /api/tasks/{id}/cancel`
- `POST /api/tasks/{id}/force-unlock`
- `GET /api/stores`
- `GET /api/stores/{id}`
- `GET /api/stores/{id}/sync-tasks`
- `POST /api/stores`
- `POST /api/stores/{id}`
- `POST /api/stores/{id}/credentials`
- `POST /api/stores/{id}/sync`
- `POST /api/stores/{id}/sync/force`
- `GET /api/dashboard/stats`
- `GET /api/dashboard/activity`
- `GET /api/dashboard/alerts`
- `GET /api/notifications`

### 5.2 前置依赖

- Task 表与状态机
- Store 主表与凭证表
- 同步 inbox / sync run 设计
- Dashboard 读侧的预聚合口径

### 5.3 开发顺序

1. `GET /api/tasks`、`GET /api/tasks/{id}`、`GET /api/tasks/{id}/events`
2. 店铺读接口
3. 店铺配置写接口
4. 凭证更新任务化接口
5. 店铺同步任务创建接口
6. Task 的 retry / cancel / force-unlock
7. Dashboard / Notification 读侧接口

### 5.4 联调顺序

1. 前端先接任务中心读侧
2. 再接店铺列表与详情
3. 再接凭证更新与同步按钮
4. 最后接 Dashboard 风险摘要和通知列表

### 5.5 验收口径

- 同店同类同步不允许并发运行
- 凭证更新返回 `task_id`
- 同步失败能通过 Task 读到状态、阶段和错误码
- Dashboard 走汇总口径，不扫热明细

### 5.6 风险点

- Task 读侧不稳定会拖垮后面所有长任务模块
- 同步幂等做错会造成重复任务与脏数据

## 6. Batch 3：履约主链

### 6.1 范围

接口：

- `GET /api/fulfillment/orders`
- `GET /api/fulfillment/orders/{id}`
- `GET /api/fulfillment/pos`
- `GET /api/fulfillment/pos/{id}`
- `GET /api/shipments/{id}`
- `POST /api/fulfillment/orders/import`
- `POST /api/fulfillment/pos`
- `POST /api/fulfillment/pos/{id}/items`
- `POST /api/fulfillment/pos/{id}/tracking`
- `POST /api/fulfillment/pos/{id}/cancel`
- `POST /api/fulfillment/orders/{id}/exception`
- `POST /api/shipments/{id}`

### 6.2 前置依赖

- 店铺与任务骨架已打通
- 订单 / 订单项 / PO / Shipment 主表已可读写
- 幂等、版本冲突、审计骨架可用

### 6.3 开发顺序

1. 订单池读接口
2. PO 列表与详情
3. 创建 PO
4. 新增 / 调整 PO 明细
5. 批量补号
6. 标记异常
7. 修改 Shipment
8. 取消 PO
9. 强制导入订单任务

### 6.4 联调顺序

1. 先接订单池列表
2. 再接 PO 详情抽屉
3. 再接创建 PO 与补号
4. 最后联调异常与取消

### 6.5 验收口径

- 禁止跨仓合单
- 已绑定订单项不能重复绑定
- 终态 PO 默认只读
- 所有高危动作有审计和幂等保护
- `If-Match-Version` 冲突能正确返回

### 6.6 风险点

- PO 写路径分叉会直接破坏履约主链
- 补号与异常流程若无版本控制，容易并发覆盖

## 7. Batch 4：竞价主链

### 7.1 范围

接口：

- `GET /api/bidding/products`
- `GET /api/bidding/products/{id}`
- `GET /api/bidding/logs`
- `GET /api/bidding/store-policies/{store_id}`
- `POST /api/bidding/floors/import`
- `POST /api/bidding/products/{id}/floor`
- `POST /api/bidding/products/{id}/enable`
- `POST /api/bidding/products/{id}/disable`
- `POST /api/bidding/stores/{id}/start`
- `POST /api/bidding/stores/{id}/stop`
- `POST /api/bidding/runs/{task_id}/cancel`

### 7.2 前置依赖

- 身份、店铺、任务、审计骨架已稳定
- `bid_products`、`autobid_store_policy`、`bid_log` 主表可用
- 护栏与错误码规则冻结

### 7.3 开发顺序

1. 商品列表与详情读接口
2. 店铺策略读接口
3. 单个底价修改
4. 单品启停
5. SKU 底价导入任务
6. 全店竞价启动 / 停止任务
7. 竞价任务取消
8. 调价日志读接口

### 7.4 联调顺序

1. 先接列表、详情、日志
2. 再接底价修改和单品启停
3. 最后接全店竞价启动 / 停止

### 7.5 验收口径

- 无人工底价不能启用
- 竞价任务必须任务化
- 调价日志完整可读
- 护栏阻断时返回明确错误码与原因

### 7.6 风险点

- 竞价启停如果不走任务化，极易失控
- 底价与调价执行链分离不清会导致误操作

---

## 8. 批次间硬依赖

```text
Batch 1
  → Batch 2
  → Batch 3
  → Batch 4
```

说明：

- `Batch 2` 依赖 `Batch 1` 的身份、权限、功能开关和审计骨架
- `Batch 3` 依赖 `Batch 2` 的任务骨架与店铺同步基础
- `Batch 4` 依赖前三批次的身份、任务、店铺、审计和错误码骨架

---

## 9. 建议联调节奏

### 第一轮

- 完成 Batch 1 的前后端联调
- 确认 `/api/auth/me`、Admin 列表、高危写接口全部稳定

### 第二轮

- 完成 Batch 2 的任务中心 + 店铺同步联调
- 确认长任务最小骨架可靠

### 第三轮

- 完成 Batch 3 的履约主链联调
- 作为 `2026-05-01` 上线主链优先验收

### 第四轮

- 完成 Batch 4 的竞价联调
- 先读侧与单品，再店铺级任务

---

## 10. 验收总口径

- 所有 `P0` 写接口都具备权限、状态前置、幂等、审计、错误码
- 所有长任务都有 `task_id`、状态、阶段、事件流和失败原因
- `super_admin / tenant_admin` 边界在所有批次中保持一致
- 履约主链在竞价主链之前完成验收
- 所有批次的联调顺序与上线 Runbook 顺序一致
