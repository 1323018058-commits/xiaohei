# P1 接口实现批次拆解

## 1. 目标

本文件把 `04_API契约逐接口展开.md` 中 `P1` 范围的接口，按“受限可用依赖链”拆成可执行的开发与联调批次。

拆解目标：

- 让 `P1` 接口从“有定义”变成“可灰度、可受限上线、可联调”
- 让 `P1` 的上线边界不反向阻断 `P0`
- 让每个批次都清楚区分：读侧、受限写侧、暂不开放能力

---

## 2. 为什么采用方案 B

采用“按受限可用依赖链拆批次”，而不是按模块平铺。

原因：

1. `Selection / Extension` 更偏轻读写，最适合作为 `P1` 首批受限能力。
2. `Warehouse` 强依赖 `Fulfillment` 主链，应该在主链稳定后补齐。
3. `Finance` 风险最高，不适合和其他 `P1` 模块一起平推，必须最后压轴。
4. `P1` 的目标不是一步到位，而是先做到“受限可用、不误导用户”。

---

## 3. 批次总览

| 批次 | 名称 | 目标 | 阻断下游 |
|---|---|---|---:|
| Batch 5 | Selection / Extension / Product Intelligence | 打通共享商品事实、利润预览、标注、导出与扩展状态页 | 否 |
| Batch 6 | Warehouse | 打通仓库操作页与 Shipment 执行流 | 否 |
| Batch 7 | Finance | 打通读侧、少量白名单重算入口与快照查看 | 否 |

---

## 4. Batch 5：Selection / Extension / Product Intelligence

### 4.1 范围

接口：

- `GET /api/selection/products`
- `POST /api/selection/products/{id}/mark`
- `POST /api/selection/export`
- `POST /api/extension/auth`
- `GET /api/extension/profile`
- `POST /api/extension/profit-preview`
- `POST /api/extension/protected-floor`
- `POST /api/extension/list-now`

### 4.2 前置依赖

- `Auth / Admin / Store` 已可用
- `library_products`、`tenant_product_guardrails`、`selection_memory`、`extension_auth_tokens` 最小结构可查
- `Task` 读侧已可用（导出任务需要回链）
- `PLID -> Takealot 商品详情` 的上游拉取链路已验证可用

### 4.3 开发顺序

1. `POST /api/extension/auth`
2. `GET /api/extension/profile`
3. `POST /api/extension/profit-preview`
4. `POST /api/extension/protected-floor`
5. `GET /api/selection/products`
6. `POST /api/selection/products/{id}/mark`
7. `POST /api/selection/export`
8. `POST /api/extension/list-now`（任务预埋版）

### 4.4 联调顺序

1. 先接 ERP 扩展授权页与 token 下发
2. 再接浏览器扩展 `PLID` 识别与详情页利润预览
3. 再接保护价保存与 `AutoBid` 同步状态
4. 再接候选商品列表与标注交互
5. 再接导出按钮与任务回链
6. 最后接 `list-now` 任务壳创建与状态跳转

### 4.5 验收口径

- 扩展 `V1` 只调用 ERP，不直接高频请求 Takealot
- 同一 `PLID` 命中共享商品事实层时可快速返回；未命中时允许后台补全并沉淀
- 成本、运费、费率只参与当前试算请求，不作为后端持久化字段
- 保护价是扩展 `V1` 唯一默认开放的持久化写动作，并且要可解释地同步到 `AutoBid`
- 候选商品列表可稳定读取
- 标注操作不要求高危审计，但要保留基本操作痕迹
- 导出走任务化，不阻塞前端
- 保护价护栏必须与共享商品事实隔离
- `list-now` 首发仅创建任务壳，不承诺复杂平台写回

### 4.6 风险点

- 若直接把导出做同步，会拖垮读接口
- 若 `library_products` 仍按 `store_id` 去重，同一商品会在多店铺下重复沉淀
- 若利润公式写在浏览器扩展里，Web 与插件会出现公式漂移
- 若把成本、运费、费率错误地写进 ERP 主库，会快速形成脏数据并误导后续利润判断
- 扩展页若展示超出实际能力的“自动写回”，会误导用户

## 5. Batch 6：Warehouse

### 5.1 范围

接口：

- `GET /api/warehouse/tasks`
- `GET /api/warehouse/batches/{id}`
- `POST /api/warehouse/shipments/{id}/receive`
- `POST /api/warehouse/shipments/{id}/pack`
- `POST /api/warehouse/shipments/{id}/label`
- `POST /api/warehouse/shipments/{id}/ship`

### 5.2 前置依赖

- `Fulfillment` 主链已可用
- `Shipment` 与 `Warehouse` 扩展表已具备最小结构
- 高危动作确认与审计规则已稳定

### 5.3 开发顺序

1. `GET /api/warehouse/tasks`
2. `GET /api/warehouse/batches/{id}`
3. `POST /api/warehouse/shipments/{id}/receive`
4. `POST /api/warehouse/shipments/{id}/pack`
5. `POST /api/warehouse/shipments/{id}/label`
6. `POST /api/warehouse/shipments/{id}/ship`

### 5.4 联调顺序

1. 先接仓库任务列表
2. 再接出库批次详情
3. 再接到仓 / 打包 / 贴标 / 出库动作

### 5.5 验收口径

- 仓库动作必须严格受当前阶段约束
- 重复扫描返回明确冲突语义
- 平板友好的主操作流可跑通
- 所有写动作可追溯到 Shipment / Batch / 操作人

### 5.6 风险点

- 若跳过阶段校验，仓库动作很容易污染履约主链
- 若扫描与出库动作没有幂等保护，会出现重复执行

## 6. Batch 7：Finance

### 6.1 范围

接口：

- `GET /api/finance/snapshots`
- `GET /api/finance/snapshots/{id}`
- `GET /api/finance/ledgers`
- `GET /api/finance/recalculation-tasks`
- `POST /api/finance/snapshots/recalculate`
- `POST /api/finance/snapshots/{id}/freeze`
- `POST /api/finance/adjustments`（不默认全面开放）

### 6.2 前置依赖

- 财务快照、流水、adjustment 基础表已具备
- `Task` 系统已可稳定承接重算任务
- `Audit` 体系已可记录高危财务动作

### 6.3 上线策略冻结

`Finance` 在 `P1` 阶段采用：

- 读侧优先
- `recalculate` 只对白名单或受限范围开放
- `adjustments` 不做默认全面开放
- `freeze` 仅 `super_admin` 可用

### 6.4 开发顺序

1. `GET /api/finance/snapshots`
2. `GET /api/finance/snapshots/{id}`
3. `GET /api/finance/ledgers`
4. `GET /api/finance/recalculation-tasks`
5. `POST /api/finance/snapshots/recalculate`
6. `POST /api/finance/snapshots/{id}/freeze`
7. `POST /api/finance/adjustments`（最后保留为受控能力）

### 6.5 联调顺序

1. 先接快照列表与详情
2. 再接流水列表
3. 再接重算任务入口与任务状态回查
4. 最后联调冻结与 adjustment 的受控路径

### 6.6 验收口径

- 读侧稳定、可读、可解释
- 重算走任务化，且仅对白名单范围开放
- `adjustments` 不是默认全面开放能力
- 财务相关高危动作必须带审计与原因

### 6.7 风险点

- 如果过早全面开放 adjustment，财务域会立刻失控
- 如果重算不任务化，会直接拖慢主站

## 7. 批次间关系

```text
Batch 5
  → Batch 6
  → Batch 7
```

说明：

- `Selection / Extension` 可以率先作为轻量 `P1` 能力开放
- `Warehouse` 建立在 `Fulfillment` 主链之上
- `Finance` 最后开放，并保持强约束

---

## 8. 与前面文档的边界关系

- `P0` 仍以 `10_P0接口实现批次拆解.md` 为主版本
- 本文件仅负责 `P1`
- `Listing / Dropship` 继续保持 `P2` 延期，不纳入本轮实现批次

## 9. 验收总口径

- `P1` 的每个接口都已明确“读侧 / 受限写侧 / 暂不开放”边界
- `Finance` 已冻结为“读侧优先 + 白名单重算 + 非默认全面 adjustment”
- `Warehouse` 已明确依赖 `Fulfillment` 主链，不反向污染 `P0`
- `Selection / Extension` 已定位为轻量受限能力，而不是主站阻断项
