# 异常处理与人工介入手册

## 1. 目标与适用范围

- 本手册是 `2026-05-01` 首发窗口及首批商用阶段的正式异常处理主版本。
- 适用对象：`operator`、`warehouse`、`tenant_admin`、`super_admin`、技术负责人、发布负责人。
- 适用范围：
  - Fulfillment / PO / 补号 / 错绑
  - Store Sync / 外部平台 / 脏数据
  - Task / Worker / Listing Worker
  - Finance 快照 / 流水 / 调整
  - 首发窗口发布期的人工介入与客户沟通
- 客户侧精简话术仍以 `docs/commercial_delivery/incident_response_playbook.md` 为对外子集；内部值班、任务恢复、权限边界和禁止动作以本手册为准。

---

## 2. 异常分级标准

### 2.1 业务影响等级

| 等级 | 定义 | 典型场景 | 是否默认阻断上线 |
|---|---|---|---:|
| `S0` | 平台级不可用、资金或大批量外部写风险 | 登录大面积失败、`super_admin` 不可用、财务误入账、重复调价 | 是 |
| `S1` | 单租户主链阻断，必须人工恢复 | 店铺同步全失败、PO 主链卡死、履约错绑 | 是 |
| `S2` | 局部任务或单店异常，系统可部分降级 | `waiting_retry`、外部平台超时、单商品上架失败 | 否 |
| `S3` | 展示异常、说明文案、非关键只读问题 | 历史 listing freshness、低优先级报表异常 | 否 |

### 2.2 人工介入权限级别

- `L1`：操作员 / 仓库执行人员可直接处理  
  仅允许执行可逆、低风险、当前租户内的业务动作，如补号、重新扫描、标记履约异常、核对原始单据。
- `L2`：`tenant_admin` 可处理  
  允许处理本租户受控恢复动作，如更新店铺凭证、触发单店重试、取消误发任务、确认店铺/订单侧人工修正。
- `L3`：必须 `super_admin`  
  允许跨租户、跨系统边界、订阅/账号生命周期、死信恢复、财务受控重算、敏感配置修正。
- `L4`：必须技术负责人 / 发布负责人联合处理  
  只用于发布日总闸门、数据库恢复、全局开关、生产 worker/scheduler、回滚与环境级风险处置。

### 2.3 A / B / C 权重冻结

- A 类主手册：`Fulfillment / PO / 补号 / 错绑`
- B 类辅助手册：`Store Sync / 外部平台 / 脏数据 / Extension / Listing Worker`
- C 类严格手册：`Finance / 重算 / adjustment / 入账`

规则：

- 第一版值班主火力优先覆盖 A 类，不平均分配篇幅。
- B 类以“恢复可观测 + 控制外部平台风险”为目标。
- C 类只允许最少数、最可审计、最不破坏历史真相的动作。

## 3. 人工介入权限矩阵

| 动作 | `L1` | `L2` | `L3` | `L4` |
|---|---|---|---|---|
| 补号、重扫、标记履约异常 | 可执行 | 复核 | - | - |
| 取消误发任务 | 不可直接调 Task API | 本租户可执行 | 跨租户可执行 | - |
| 立即重试 `waiting_retry / manual_intervention` 任务 | 不可直接调 Task API | 本租户可执行 | 死信/跨租户可执行 | - |
| 更新店铺凭证并复测 | - | 本租户可执行 | 跨租户或平台级复核 | - |
| 强制全量同步 | - | 本租户可执行 | 跨租户可执行 | 发布期需确认 |
| 履约错绑修正 / PO 人工覆盖 | 仅收集证据 | 本租户受控处理 | 高风险或跨对象修正 | 发布期主链异常联合处理 |
| 财务快照重算 | - | 不默认开放 | 受控执行 | 发布期或事故期联合确认 |
| 财务 adjustment / 流水修正 | - | 禁止 | 受控执行 | 大额或整批风险联合确认 |
| 租户恢复 / 停用 / 重置管理员密码 | - | 禁止 | 可执行 | 大面积风险联合确认 |
| 数据库恢复 / 全局开关 / 回滚 | - | - | - | 仅联合处理 |

补充：

- 当前任务 API `POST /api/tasks/{task_id}/retry-now` 与 `POST /api/tasks/{task_id}/cancel` 仅对 `super_admin`、`tenant_admin` 开放。
- `operator`、`warehouse` 当前只能通过业务页动作、单据核对和人工确认流程参与恢复，不直接操作任务总线。

## 4. 异常分类总表

| 类别 | 典型状态 / 错误码 | 默认责任人 | 默认级别 | 第一动作 |
|---|---|---|---|---|
| A1 履约错绑 | `FULFILL_ORDER_ALREADY_BOUND` | 履约运营 | `L2` | 冻结继续绑定，先核对订单项与 PO |
| A2 跨仓合单被拦 | `FULFILL_CROSS_WAREHOUSE_PO_DENIED` | 履约运营 | `L2` | 拆单，不做跨仓强绑 |
| A3 补号 / 扫描异常 | `WAREHOUSE_SCAN_DUPLICATED`、`WAREHOUSE_OPERATION_INVALID_STAGE` | 仓库 | `L1` | 对实物与系统记录，停止重复扫描 |
| A4 履约人工覆盖 | `manual_intervention`、`dead_letter` | 履约负责人 | `L2/L3` | 看 task 详情、事件、审计，确认是否重试 |
| B1 店铺鉴权失败 | `STORE_AUTH_FAILED`、`STORE_CREDENTIAL_INVALID` | 店铺管理员 | `L2` | 先换凭证，不先怪 Takealot |
| B2 外部平台抖动 | `STORE_PLATFORM_UNAVAILABLE`、`EXT_TIMEOUT`、`EXT_RATE_LIMITED` | 技术值班 | `L2/L3` | 观察重试，不先人工硬刷 |
| B3 脏数据隔离 | `quarantined`、`EXT_DIRTY_DATA_QUARANTINED` | 运营 + 技术 | `L3` | 保留原始 payload，禁止手改主表 |
| B4 上架链卡住 | `LISTING_*`、`manual_intervention`、`waiting_listing_worker` | 上架运营 | `L2/L3` | 先查保护价、条码、售价、店铺凭证 |
| B5 订阅导致写阻断 | `past_due`、`402 Payment Required` | 商务 / 超管 | `L3` | 先看订阅，不先排平台 |
| C1 财务快照过期 | `FINANCE_SNAPSHOT_STALE` | 财务负责人 | `L3` | 受控重算，不改历史流水 |
| C2 财务流水不可写 | `FINANCE_LEDGER_IMMUTABLE` | 财务负责人 | `L3` | 走 adjustment，不覆盖旧记录 |

## 5. A 类主手册：Fulfillment / PO / 补号 / 错绑

### A-1 订单已绑定错误 PO

- 触发条件：出现 `FULFILL_ORDER_ALREADY_BOUND`，或运营确认订单项被绑定到错误 PO。
- 用户可见症状：订单无法再次绑定、PO 工作台状态与实际处理对象不一致。
- 系统自动处理：阻止重复绑定，保留当前绑定真相，不自动改绑。
- 人工处理动作：
  - 先到订单详情、PO 工作台、审计日志确认真实归属。
  - 暂停该订单项后续人工处理，避免继续补号或出库。
  - 若仅是页面读侧缓存问题，刷新任务与详情即可，不做对象修正。
  - 若确认真错绑，走受审计的履约人工覆盖/绑定修正流程；若当前没有专用后台按钮，升级 `L3` 走受控修正。
- 处理权限级别：默认 `L2`，跨店铺、跨租户或需修正历史关系时升 `L3`。
- 确认人：履约负责人 + 当前租户管理员。
- 审计要求：必须记录 `request_id`、错误订单项、原 PO、目标 PO、原因、确认人。
- 后置验证：订单池、PO 工作台、任务中心、审计页四处口径一致。
- 禁止动作：禁止直接改数据库绑定关系；禁止边核对边继续打包。
- 是否阻断上线：若为单笔局部异常，否；若连续出现或影响主链批量出库，是。
- 客户话术：  
  > 我们已先冻结这笔订单的后续处理，正在核对它当前绑定的采购单与实际归属，确认后会以受审计方式修正。

### A-2 跨仓合单被系统拦截

- 触发条件：出现 `FULFILL_CROSS_WAREHOUSE_PO_DENIED`。
- 用户可见症状：创建 PO 或追加明细失败，提示禁止跨仓合单。
- 系统自动处理：拒绝写入，不产生跨仓脏 PO。
- 人工处理动作：
  - 确认订单项真实仓库归属。
  - 按仓库拆分 PO，不再尝试强行并单。
  - 如运营认为业务必须合并，升级 `L3` 评估是否是主数据仓库配置错误。
- 处理权限级别：`L2`
- 确认人：履约负责人
- 审计要求：保留原请求、被拦截原因、最终拆分方案。
- 后置验证：拆分后每个 PO 仅含单仓商品，后续补号/出库可继续。
- 禁止动作：禁止通过线下 Excel 拼单后再手工补录为单 PO。
- 是否阻断上线：单笔不阻断；若仓库主数据普遍错误，阻断。
- 客户话术：  
  > 系统拦截的是跨仓合并风险，不是订单丢失。我们会按仓库拆成正确采购单，保证后续履约可追踪。

### A-3 补号缺失、重复或错绑

- 触发条件：补号失败、物流号重复、扫描命中 `WAREHOUSE_SCAN_DUPLICATED`。
- 用户可见症状：无法继续出库、重复扫描提示、运单记录与实物不一致。
- 系统自动处理：阻止重复写入，保留最近一次有效记录。
- 人工处理动作：
  - `L1` 先核对包裹标签、订单号、PO、扫描时间。
  - 对于重复扫描，先确认是否已成功落库，避免二次补号。
  - 对于错绑，升级 `L2` 由租户管理员确认需不需要撤销并重新绑定。
  - 对于已出库对象，只允许追加异常说明，不允许静默覆盖物流号。
- 处理权限级别：`L1` 收集与核对，`L2` 执行修正。
- 确认人：仓库班长 / 履约负责人。
- 审计要求：记录原物流号、目标物流号、包裹号、操作人、原因。
- 后置验证：Shipment、订单详情、履约异常记录一致。
- 禁止动作：禁止口头确认后直接覆盖物流号；禁止跳过异常说明。
- 是否阻断上线：若影响整批出库节奏，是；单笔局部异常，否。
- 客户话术：  
  > 我们正在核对这笔包裹的物流号与系统记录，确认前不会继续推进出库，以避免后续追踪错乱。

### A-4 仓库动作与当前阶段不匹配

- 触发条件：出现 `WAREHOUSE_OPERATION_INVALID_STAGE`。
- 用户可见症状：当前页面不允许打包、贴标、出库或补号。
- 系统自动处理：拒绝该动作，保持对象只读或停留在当前合法阶段。
- 人工处理动作：
  - 先看订单、PO、Shipment 当前 `status + stage`。
  - 确认是否有上一步缺失，例如未到仓就出库、未打包就贴标。
  - 若只是任务延迟，等待或重试上一步任务；若阶段真错乱，升级 `L2/L3`。
- 处理权限级别：`L1` 核对，`L2` 修正，必要时 `L3`。
- 确认人：仓库负责人。
- 审计要求：记录原阶段、阻断动作、最终恢复路径。
- 后置验证：下一合法动作可执行，审计可回看。
- 禁止动作：禁止通过线下跳步后再回填系统。
- 是否阻断上线：整仓大面积出现时阻断。
- 客户话术：  
  > 当前系统拦截的是阶段顺序错误，不会丢单。我们会先恢复正确步骤，再继续仓库操作。

## 6. B 类辅助手册：Store Sync / 外部平台 / 脏数据

### B-1 店铺鉴权失败

- 触发条件：`STORE_AUTH_FAILED`、`STORE_CREDENTIAL_INVALID`、店铺凭证校验失败。
- 用户可见症状：店铺同步失败、店铺 freshness 过期、同步任务停在失败/手动介入。
- 系统自动处理：记录 `error_code`、审计 `store.sync.worker` / `store.credentials.update`，不继续盲重试无效凭证。
- 人工处理动作：
  - 先核对订阅是否正常，再核对凭证，不先归因平台故障。
  - 要求客户通过安全通道重发有效 Key/Secret。
  - 更新凭证后只做一次受控复测，再观察任务状态。
- 处理权限级别：`L2`；跨租户或租户生命周期问题用 `L3`。
- 确认人：店铺管理员 / 超管。
- 审计要求：凭证更新必须写审计；前端仅展示脱敏值。
- 后置验证：凭证状态恢复 `valid / configured / validating`，同步任务成功。
- 禁止动作：禁止把明文凭证发到普通群聊；禁止反复强制同步刷平台。
- 是否阻断上线：若是唯一生产店铺，阻断；测试残留或非生产店铺，可不阻断。
- 客户话术：  
  > 当前这组平台凭证没有通过鉴权。请通过安全通道发送新的生产凭证，我们会立即重新校验并做一次受控同步。

### B-2 外部平台超时、限流或临时不可用

- 触发条件：`STORE_PLATFORM_UNAVAILABLE`、`EXT_TIMEOUT`、`EXT_RATE_LIMITED`、`ORDER_PLATFORM_UNAVAILABLE`。
- 用户可见症状：任务进入 `waiting_retry`，多次失败后进入 `manual_intervention` 或 `dead_letter`。
- 系统自动处理：指数退避重试，保留 `next_retry_at`、错误详情、任务事件。
- 人工处理动作：
  - 先看是否还在自动重试窗口内。
  - 若仍在 `waiting_retry`，优先等待，不立刻人工硬刷。
  - 若超过预期窗口或转 `manual_intervention`，由 `L2` 触发一次 `retry-now`。
  - 若出现平台级连续异常，升级 `L4` 决定是否降级或延后发布。
- 处理权限级别：默认 `L2`，批量或上线期异常升 `L4`。
- 确认人：技术值班 / 发布负责人。
- 审计要求：保留 `task.retry_requested`、`task.cancel`、相关 `store.sync.*` 审计。
- 后置验证：重试后任务成功，或明确进入可接受降级态。
- 禁止动作：禁止连续手工重试形成二次打爆；禁止未核对 `next_retry_at` 就取消任务。
- 是否阻断上线：若是真实生产店铺连续异常且影响主链，阻断。
- 客户话术：  
  > 当前是平台接口瞬时不可用，我们已切到受控重试和观察模式；确认恢复后会再推进同步，不会盲目连续提交。

### B-3 脏数据或隔离态

- 触发条件：`EXT_DIRTY_DATA_QUARANTINED`、任务/对象进入 `quarantined`。
- 用户可见症状：商品、订单或 listing 不可继续处理，页面提示需要人工介入。
- 系统自动处理：隔离对象，保留原始 payload，不继续写主表真相。
- 人工处理动作：
  - 导出原始 payload、请求链路、任务事件。
  - 判断是外部字段缺失、类型脏、重复对象还是内部映射规则缺失。
  - 修正规则优先，最后才处理单对象。
  - 只有确认规则收口后才允许重试。
- 处理权限级别：`L3`
- 确认人：技术负责人。
- 审计要求：记录原始对象标识、隔离原因、恢复理由。
- 后置验证：对象离开隔离态且无重复写入。
- 禁止动作：禁止直接改主库对象“假装正常”；禁止删除原始 payload。
- 是否阻断上线：若为批量生产数据，阻断。
- 客户话术：  
  > 当前是上游返回的数据不满足系统校验，我们已隔离这批异常输入，避免污染正式数据，修正规则后再恢复处理。

### B-4 Extension / Listing Worker 上架链卡住

- 触发条件：`LISTING_WORKER_DISABLED`、`LISTING_BARCODE_MISSING`、`LISTING_SELLING_PRICE_MISSING`、`LISTING_BATCH_PENDING`、`manual_intervention`。
- 用户可见症状：`protected-floor` 已保存，但 `list-now` 卡住；`listing_job` 不进入 `ready_to_submit`。
- 系统自动处理：创建 `EXTENSION_LIST_NOW`、`listing_job`、`PROCESS_LISTING_JOB` 任务；必要时转 `waiting_retry` 或 `manual_intervention`。
- 人工处理动作：
  - 先查 `protected-floor` 是否已保存成功。
  - 再查 `platform_product_id / barcode / selling_price / store credentials` 是否齐全。
  - `LISTING_BATCH_PENDING` 优先等待批次轮询，不先手工重复创建报价。
  - `LISTING_BARCODE_MISSING` 或 `LISTING_SELLING_PRICE_MISSING` 必须先补输入，再允许重试。
  - 仅 `tenant_admin / super_admin` 可通过任务 API 做 `retry-now` / `cancel`。
- 处理权限级别：`L2`；需要修规则或批量平台核对时升 `L3`。
- 确认人：上架运营负责人。
- 审计要求：保留 `listing_job`、`task.manual_intervention`、`task.retry_requested`、上架审计。
- 后置验证：`list_now_task_status = succeeded`、`listing_job_status = ready_to_submit` 或明确 `manual_intervention` 原因。
- 禁止动作：禁止未确认批次结果就重复发起 `list-now`；禁止缺少条码时强行绕过官方上架路径。
- 是否阻断上线：若扩展是受限能力，一般不阻断主站；若首发演示依赖该链路，可局部阻断。
- 客户话术：  
  > 当前上架链已经接到任务，但在平台批次确认/必要字段校验阶段被安全护栏拦住。我们会先补齐条件或等待批次回执，再继续推进。

### B-5 `past_due` / `402` 导致写操作被阻断

- 触发条件：Dashboard 显示 `past_due`，创建店铺或同步返回 `402 Payment Required`。
- 用户可见症状：系统可读但不可写，运营误以为是平台故障。
- 系统自动处理：写入拦截，不继续向外部平台发请求。
- 人工处理动作：
  - 先查 `trial_ends_at` / `current_period_ends_at`。
  - 已确认续费则由 `super_admin` 更新订阅周期。
  - 更新后复核 `/api/auth/me`、Dashboard、目标写操作是否恢复。
- 处理权限级别：`L3`
- 确认人：商务负责人 / 超管。
- 审计要求：订阅修改必须写审计。
- 后置验证：状态恢复 `active`，原写路径可执行。
- 禁止动作：禁止把 `402` 当成平台接口故障去排查平台。
- 是否阻断上线：若首批付费租户真实受影响，阻断其交付。
- 客户话术：  
  > 当前不是平台接口异常，而是工作区处于待续费状态，系统因此暂停了写操作。我们正在补齐订阅周期并恢复可写状态。

## 7. C 类严格手册：Finance

### C-1 利润快照过期或需要重算

- 触发条件：`FINANCE_SNAPSHOT_STALE`，或业务确认快照时点已失真。
- 用户可见症状：页面显示快照过期、利润数字与最新订单/费用不同步。
- 系统自动处理：拒绝把旧快照当最新真相，提示重算。
- 人工处理动作：
  - 先确认是否属于白名单或允许窗口内的受控重算。
  - 由 `super_admin` 发起一次任务化重算。
  - 重算后确认版本、时间戳、关联流水仍可追溯。
- 处理权限级别：`L3`
- 确认人：财务负责人。
- 审计要求：保留 `finance.snapshot.recalculate` 审计和 task 链路。
- 后置验证：新快照生成，旧快照不被物理覆盖。
- 禁止动作：禁止手工改快照表；禁止无任务、无审计直接改利润数。
- 是否阻断上线：若影响对外结算或大面积经营判断，是。
- 客户话术：  
  > 当前利润快照需要按最新流水重算。我们会通过受控任务重新生成快照，不会直接修改历史记录。

### C-2 财务流水不可覆盖

- 触发条件：`FINANCE_LEDGER_IMMUTABLE`，或需要修正历史流水。
- 用户可见症状：直接修改旧流水被系统拒绝。
- 系统自动处理：拒绝覆盖写，要求走 adjustment。
- 人工处理动作：
  - 确认是否真需要修正历史财务结果。
  - 由 `super_admin` 按 adjustment 方式新增修正记录。
  - 如需重算快照，串联到 C-1。
- 处理权限级别：`L3`
- 确认人：财务负责人 + 超管。
- 审计要求：必须记录原流水、adjustment 原因、影响范围。
- 后置验证：原流水保持不可变，新 adjustment 可追溯。
- 禁止动作：禁止物理删除或覆盖流水；禁止为追求“对账好看”直接改旧记录。
- 是否阻断上线：若已产生错误入账或金额错乱，是。
- 客户话术：  
  > 财务流水是不可直接覆盖的。我们会通过可追溯的调整记录修正结果，确保历史账目仍然可审计。

### C-3 财务人工处理总规则

- Finance 只允许“重算、调整、冻结、替代快照”这类受控动作。
- 不允许以“紧急”为由直接改数据库主表。
- 不允许跳过审计与原因说明。
- 涉及资金、对账、发票、结算口径的异常，默认按 `S0/S1` 处理。

## 8. Task / ErrorCode / Audit 映射

### 8.1 任务状态到人工动作

| Task 状态 | 默认含义 | 允许人工动作 | 默认级别 |
|---|---|---|---|
| `queued` / `leased` / `running` | 已入队或正在执行 | 误发时可 `cancel` | `L2` |
| `waiting_retry` | 系统已安排自动重试 | 先等待；必要时 `retry-now` | `L2` |
| `failed_retryable` / `timed_out` | 暂时失败，可恢复 | `retry-now` | `L2` |
| `manual_intervention` | 系统要求人工判断 | 先查详情/事件，再决定 `retry-now` 或业务修正 | `L2/L3` |
| `dead_letter` | 多次失败或不可直接恢复 | 只允许受控恢复 | `L3` |
| `cancelled` | 已取消 | 如业务确认可重新进入，可 `retry-now` | `L2/L3` |
| `quarantined` | 数据或风险被隔离 | 先修规则/输入，不直接重试 | `L3` |

补充：

- 任务读接口：`GET /api/tasks`、`GET /api/tasks/{task_id}`、`GET /api/tasks/{task_id}/events`
- 任务写接口：`POST /api/tasks/{task_id}/retry-now`、`POST /api/tasks/{task_id}/cancel`
- 当前任务写接口仅 `super_admin` 与 `tenant_admin` 可用。

### 8.2 常见错误码到恢复动作

| 错误码 / 家族 | 典型域 | 默认动作 | 必须看的审计 / 事件 |
|---|---|---|---|
| `TASK_CANCELLED` | Task / Store / Orders / Listing | 确认是否误取消，必要时重新入队 | `task.cancel`、`task.cancelled` |
| `TASK_HEARTBEAT_STALE` | Worker | 查 worker heartbeat、host check、是否自动重入 `waiting_retry` | `task.failed.retryable`、`task.retry_scheduled` |
| `STORE_AUTH_FAILED` | Store Sync | 更新凭证后受控复测 | `store.credentials.update`、`store.sync.worker` |
| `STORE_PLATFORM_UNAVAILABLE` | Store Sync | 先等自动重试，再决定人工重试 | `task.retry_scheduled`、`store.sync.worker` |
| `LISTING_BARCODE_MISSING` | Listing Worker | 先补条码，不直接重试 | `task.manual_intervention`、`listing.job.*` |
| `LISTING_SELLING_PRICE_MISSING` | Listing Worker | 先补售价，不直接重试 | `task.manual_intervention`、`listing.job.*` |
| `LISTING_BATCH_PENDING` | Listing Worker | 等批次轮询，不重复发起上架 | `task.retry_scheduled` |
| `FULFILL_ORDER_ALREADY_BOUND` | Fulfillment | 冻结后续动作，核对订单项与 PO | `po.bind_order_item`、履约人工覆盖审计 |
| `WAREHOUSE_SCAN_DUPLICATED` | Warehouse | 核对是否已成功扫描落库 | 仓库操作审计 |
| `FINANCE_SNAPSHOT_STALE` | Finance | 受控重算 | `finance.snapshot.recalculate` |

### 8.3 审计最少要求

所有人工介入至少保留：

- `request_id`
- `actor_user_id`
- `actor_role`
- `target_type`
- `target_id`
- `action`
- `before`
- `after`
- `reason`
- `result`
- `error_code`（如失败）
- `task_id`（如与任务有关）

## 9. 上线期特殊处理规则

### 9.1 发布日总规则

- 发布日先跑：
  - `npm run ops:env:check`
  - `npm run db:backup`
  - `npm run db:restore:check`
  - `npm run release:preflight`
  - `npm run ops:guardrails:strict`
  - `npm run host:check`
- `release:preflight` 是上线前总闸门。
- `host:check` 通过前，不宣布进入正式放量。

### 9.2 哪些 warning 可以接受

仅在 `L4` 联合确认后，以下 warning 才能被接受：

- `store_freshness`
- `listing_health`
- `sync_failure_budget`

前提必须同时满足：

- 明确是测试残留、旧样本、非公网前置准备项
- 不是真实生产店铺
- 不影响当前首发主链

### 9.3 哪些情况必须立刻阻断或回滚

- 登录大面积失败
- `super_admin` 后台不可用
- worker 心跳持续陈旧
- 同步异常拖垮主站
- 重复财务入账
- 重复调价
- PO 主链错绑批量发生
- 监控 / 审计 / 备份链路不可用

### 9.4 只能 `L4` 处理的上线动作

- 切换 `XH_SESSION_COOKIE_SECURE`
- 切换 `XH_DB_BOOTSTRAP_DEMO_DATA`
- 配置 / 替换公网 webhook 与 alert webhook
- 安装 / 修改 `XiaoheiERPWorker` 与 `XiaoheiERPOpsGuardrails`
- 应用回滚与数据库恢复

## 10. TOP 10 高频异常速查表

| 场景 | 第一检查项 | 第一动作 | 默认级别 | 对客户一句话 |
|---|---|---|---|---|
| 客户无法登录 | 租户状态 / 用户状态 / 密码重置 | 先查租户与账号，不先怪平台 | `L3` | 我们先确认是工作区状态还是账号问题 |
| Dashboard `past_due` | `trial_ends_at` / `current_period_ends_at` | 先改订阅，再复测写路径 | `L3` | 当前是订阅写保护，不是平台故障 |
| 创建店铺/同步返回 `402` | 订阅状态 | 不排查 Takealot，先恢复订阅 | `L3` | 系统正在恢复工作区可写状态 |
| 店铺鉴权失败 | Key/Secret 来源 | 更新凭证后只做一次受控复测 | `L2` | 请通过安全通道提供新凭证 |
| 同步成功但没有 listing | 正确店铺 / 上游是否有商品 | 查任务事件与筛选条件 | `L2` | 连接已建立，正在确认店铺与商品范围 |
| `list-now` 卡住 | 条码 / 售价 / 保护价 / worker 开关 | 先补条件，不重复发起 | `L2` | 上架任务已接收，正在补齐必要条件 |
| `LISTING_BATCH_PENDING` | 批次状态 | 等待轮询，不手工重复上架 | `L2` | 我们在等平台批次回执 |
| 订单已绑定错误 PO | 订单项与 PO 归属 | 先冻结处理，后走受控修正 | `L2/L3` | 我们先核对绑定关系，再修正 |
| 重复扫描/补号异常 | 实物标签与系统记录 | 停止重复扫描，先核对 | `L1/L2` | 我们在核对包裹与物流号真相 |
| 财务快照过期 | 最新流水 / 快照版本 | 走受控重算，不改历史流水 | `L3` | 我们会重算快照，不会直接改旧账 |

## 11. 关键冻结点

- 禁止无 `request_id`、无 `reason` 的高危人工介入。
- 禁止 `operator` / `warehouse` 直接操作任务总线。
- 禁止把 `402`、`past_due` 误判成平台故障。
- 禁止未核对 `next_retry_at` 就连续人工重试。
- 禁止未确认批次结果就重复发起 `list-now` 或平台写入。
- 禁止直接改数据库核心表修 PO 绑定、listing、财务流水。
- 禁止物理覆盖或删除财务历史。
- 禁止在上线窗口由单人修改全局环境开关、worker、scheduler。
- 禁止在客户沟通中承诺未经确认的外部平台恢复时间。
- 本手册与 `附录C_错误码表.md`、`附录G_任务系统字段与状态全集.md`、`附录I_审计字段与高危动作清单.md`、`08_上线_Runbook.md` 一起构成首发值班标准口径。
