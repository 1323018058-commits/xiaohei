# Program Implementation Preparation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Freeze the engineering rollout order before coding starts, including the first production slice, cross-team sequencing, and dependency gates.

**Architecture:** This document transitions the ERP project from document-governance mode into engineering-preparation mode. It defines implementation phases, the first minimal control-plane slice, dependency rules between backend/frontend/schema/worker workstreams, and the order in which business domains are allowed to start.

**Tech Stack:** Next.js, Python backend, PostgreSQL, Redis, OpenAPI-generated TS types, modular monolith

---

## 1. 目标

本文件用于在真正开始写程序前，冻结工程推进顺序，避免多个域同时开工导致返工。

## 2. 第一批开工范围

首批实际开工模块固定为“控制面最小闭环”：

1. `Auth`
2. `Admin`
3. `Task 骨架`
4. `Store 基础读写`

原因：

- 权限、角色、功能开关必须先稳定
- 长任务骨架必须先统一
- 店铺是第一个真实业务入口
- 后续 `Fulfillment / AutoBid / Finance` 都依赖这套底座

## 3. 工程阶段总排表

| Phase | 名称 | 目标 | 允许启动的工作流 | 阻断条件 |
|---|---|---|---|---|
| Phase 0 | 工程基建 | 冻结目录、启动链、契约链 | Backend 基础、Frontend 基础、DB 基础 | 无 |
| Phase 1 | 控制面最小闭环 | 打通登录、Admin、Task、Store | Auth/Admin/Task/Store | OpenAPI 生成链未通 |
| Phase 2 | Fulfillment 主链 | 打通订单池、PO、补号、异常 | Fulfillment 前后端 + DDL | Task / Store 未稳定 |
| Phase 3 | AutoBid 主链 | 打通底价、单品启停、全店竞价 | Bidding 前后端 + DDL | Task / Audit / Store 未稳定 |
| Phase 4 | P1 受限可用域 | 受限开放 Selection / Warehouse / Finance | Selection/Warehouse/Finance | P0 主链未稳定 |
| Phase 5 | P2 延后域 | Listing / Dropship 闭环补齐 | Listing / Dropship | 首发版本未稳定 |

## 4. Phase 0 必须完成项

- `OpenAPI -> TS Types` 生成链可运行
- `apps/api/api_main.py` 与 `apps/api/worker_main.py` 启动链明确
- 前端 `(auth)` 与 `(dashboard)` 物理隔离
- 系统级总控表（`system_settings` 或 `global_feature_flags`）写入 DDL 计划
- `DangerousActionDialog` 作为前端基建件列入 Wave 1

## 5. Phase 1 切片顺序

### Slice 1：登录态闭环

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`
- 前端 `login` 页面
- Dashboard 顶层用户状态读取

### Slice 2：Admin 用户流

- 用户列表 / 详情
- 功能开关修改
- 重置密码 / 禁用 / 启用 / 强制下线
- 审计摘要

### Slice 3：Task 读侧

- `GET /api/tasks`
- `GET /api/tasks/{id}`
- `GET /api/tasks/{id}/events`
- 统一任务入口 UI

### Slice 4：Store 基础流

- 店铺列表 / 详情
- 创建 / 更新店铺
- 更新凭证
- 触发同步任务

## 6. 前后端协同规则

- 后端先提供 OpenAPI 契约，再允许前端接入
- 前端禁止在契约未生成前手写接口返回类型
- 高危动作页面必须先接入 `DangerousActionDialog`
- 长任务页面必须先接入统一 Task 读侧

## 7. 数据库协同规则

- DDL 先走 Batch 1 / Batch 2，再允许 Phase 1 编码
- `system_settings` / `global_feature_flags` 属于控制面基础能力，应在 Phase 0/1 前完成设计
- 所有高危写路径依赖审计落表可用

## 8. 进入程序实现的准入条件

满足以下条件，视为可以真正开始程序实现：

- 目录结构冻结
- 契约生成链冻结
- Worker 入口策略冻结
- Auth / Dashboard 路由组边界冻结
- 首批开工模块冻结
- 文档收口第一版完成
