# 前端风格与交互设计规范

## 1. 设计定位

ProfitLens / 小黑 ERP 的前端应采用极简、清爽、低学习成本的现代商业工具风格。

它不是传统 ERP 的拥挤表格，也不是营销站，而是一个让用户“少思考、快操作、少犯错”的任务型工作台。

设计关键词：

- 极简主义
- 呼吸感
- 高信噪比
- 傻瓜式流程
- 低视觉疲劳
- 状态清晰
- 高风险动作可控

## 2. 视觉语言：极简主义与呼吸感

### 2.1 空间与留白

- 摒弃传统 ERP 拥挤、密集、压迫感强的表格布局
- 增加模块之间、卡片之间、表单字段之间的留白
- 让用户第一眼聚焦于关键数据和下一步动作
- 默认不把所有字段一次性塞给用户
- 通过布局层级减少视觉疲劳，而不是依靠颜色堆叠信息

### 2.2 色彩体系

- 主背景使用浅灰色或纯白
- 主要内容区使用白色卡片
- 主操作色采用 Apple 风格蓝色，接近 San Francisco Blue
- 状态颜色使用低饱和度语义色
- 危险操作使用克制但明确的红色
- 禁止使用高饱和、大面积、强对比的廉价后台色彩

建议语义：

- 蓝色：主操作、当前选中、关键路径
- 绿色：成功、已完成、健康
- 橙色：待处理、警告、即将到期
- 红色：失败、异常、禁用、调价风险
- 灰色：只读、禁用、历史状态

### 2.3 卡片式布局

- 核心任务以圆角白色卡片承载
- 卡片用于封装一个明确任务或一组强相关信息
- 卡片内必须有清晰标题、关键指标、状态和主操作
- 卡片布局优先兼容平板与移动端

适合卡片化的对象：

- 自动竞价设置
- 店铺健康
- 待处理订单
- PO 摘要
- 任务进度
- 账号到期提醒
- 系统风险提示

### 2.4 字体与图标

- 优先使用系统原生字体
- macOS 使用 San Francisco
- Windows 使用 Segoe UI
- 通过字号、字重、间距建立信息层级
- 不依赖大量颜色区分主次
- 图标使用细线条 Outline Icons
- 图标只辅助识别，不替代文字说明

## 3. 全局布局

### 3.1 顶层结构

```text
极简侧边导航 / 顶部任务导航
  + 顶部系统状态条
  + 主任务工作区
  + 右侧抽屉 / 轻量弹层
```

### 3.2 导航原则

- 导航必须任务导向，而不是数据库表导向
- 一级导航只保留最高频模块
- 二级功能通过抽屉式菜单、页面内 Tab 或局部操作展开
- 用户应始终停留在当前任务流中，不被多层页面跳转打断

首发一级导航建议：

- Dashboard
- Selection
- Listing
- Bidding
- Fulfillment
- Finance
- Admin

可收纳到二级或抽屉的能力：

- Store 设置
- Warehouse 操作
- Extension 设置
- System Health
- Audit Logs

### 3.3 顶部状态条

顶部状态条展示：

- 当前店铺
- 当前用户
- 订阅状态
- 系统健康
- 后台任务进度
- 通知入口

顶部状态条必须克制，不得压迫主工作区。

## 4. 交互逻辑：以效率为中心的傻瓜式流程

### 4.1 智能缺省与预填

系统应尽量减少用户输入。

必须优先设计：

- 根据历史记录预填常用字段
- 根据关键词自动推荐类目
- 根据店铺和目的仓自动筛选可合单订单
- 根据历史物流信息预填常用承运信息
- 根据商品与 BuyBox 数据提示合理竞价区间

用户理想操作路径应接近：

```text
系统推荐
  → 用户确认
  → 系统执行
  → 用户查看结果
```

### 4.2 微弱但明确的反馈

- 保存成功时，按钮可短暂转为绿色对勾
- 轻量成功反馈使用 toast，不使用大遮罩
- 高风险失败必须在页面内明确展示
- 长任务开始后，必须给出任务入口
- 避免用巨大弹窗打断用户工作流

### 4.3 抽屉式流程

优先使用右侧抽屉承载：

- 详情查看
- 快速编辑
- 审计记录
- 任务日志
- 异常原因
- 二级配置

抽屉可以让用户不离开当前列表或看板，保持任务上下文。

### 4.4 高风险动作

以下动作必须二次确认：

- 重置密码
- 禁用账号
- 修改到期时间
- 修改功能权限
- 启动 / 停止全店自动竞价
- 批量调价
- 取消 PO
- 标记订单异常
- 财务调整

二次确认必须展示：

- 操作对象
- 影响范围
- 是否可恢复
- 操作原因

## 5. 数据呈现：高信噪比表格

### 5.1 动态列管理

- 默认只展示最核心的 5 到 6 个字段
- 高级字段默认隐藏
- 用户可切换高级视图
- 用户可自定义列显示
- 系统必须记住用户的列偏好

默认表格优先展示：

- 图片
- SKU / 订单号
- 核心价格或金额
- 当前状态
- 下一步动作
- 异常原因

### 5.2 表格行为

- 默认服务端分页
- 默认服务端筛选
- 搜索必须防抖
- 排序由服务端完成
- 大表不得一次性全量加载
- 批量操作必须二次确认

### 5.3 视觉化对账

履约中心必须优先使用视觉对比降低误操作。

设计要求：

- 商品图片与物流单号并列显示
- 商品图片与订单信息同屏展示
- PO 内逐单补号时必须能看到商品图
- 仓库操作时优先展示图片、SKU、数量和目的仓
- 防错优先级高于字段完整性

### 5.4 进度感知

耗时任务不得只显示 spinner。

必须展示：

- 顶部细进度条
- 当前阶段
- 已完成数量
- 失败数量
- 最近事件
- 可取消 / 可重试入口

适用场景：

- 全站采集
- 店铺全量同步
- 大规模调价
- 批量铺货
- submission / review 同步

## 6. 关键模块设计亮点

### 6.1 AutoBid 自动竞价工作台

视觉气质参考金融交易软件，但保持极简。

核心设计：

- 用简洁折线图展示 BuyBox 波动
- 用区间带展示 floor / ceiling 安全边界
- 当前价格、BuyBox、底价、上限必须同屏可见
- 底价调整可使用 Slider，但必须允许精确输入
- Slider 调整不得直接提交，必须点击确认保存
- 调价日志以时间线形式展示
- 跳过、失败、护栏拦截必须清楚展示原因

### 6.2 Listing / Dropship 铺货流水线

采用看板模式表达流程。

看板列建议：

- 已抓取
- AI 改写中
- 待确认
- loadsheet 已生成
- 待提交
- 审核中
- 通过
- 驳回 / 需返修

交互要求：

- 支持批量处理
- 支持一键确认系统推荐
- 支持查看 AI 改写前后对比
- 支持驳回返修与重新提交
- 拖拽只能改变内部工作流状态，不得绕过后端状态机

### 6.3 Fulfillment 履约中心

履约页面不是订单表，而是防错工作台。

核心设计：

- 订单池按目的仓聚合
- 同仓可合单订单突出显示
- 商品图片、数量、SKU、物流单号并列
- PO 使用右侧详情抽屉
- 逐单补号时必须保留订单上下文
- 异常单必须可一眼识别

### 6.4 Admin 管理控制台

Admin 页面必须极简但严肃。

核心设计：

- 用户列表默认显示账号、角色、状态、到期时间、功能权限摘要
- 重置密码、禁用、修改到期时间等高危动作进入抽屉
- 权限配置使用清晰开关与套餐模板
- 审计日志以时间线展示
- 系统健康使用卡片展示，不做复杂 BI

## 7. 页面优先级

### `P0`

- Login
- Dashboard
- Admin
- Bidding
- Fulfillment

### `P1`

- Store
- Selection
- Finance
- Warehouse
- Extension 入口页

### `P2`

- Listing / Dropship 完整看板工作台

## 8. 响应式策略

首发仍以桌面端为主，但卡片式布局必须天然兼容平板。

- 桌面端：完整工作台能力
- 平板端：查看、确认、轻量编辑
- 手机端：登录、查看关键状态、处理少量确认动作
- 仓库扫码场景优先适配移动宽度

## 9. 性能与 1000 用户并发前端约束

- 列表必须分页
- 搜索必须防抖
- Dashboard 必须读汇总接口
- 前端不得高频轮询热接口
- 长任务进度必须退避轮询或使用事件流
- 同一页面不得重复触发相同写请求
- 用户多标签页操作必须依赖后端幂等保护
- 动画必须克制，不能影响低性能设备操作

## 10. 验收标准

- 页面第一眼能看到关键状态和下一步动作
- 默认视图不超过 5 到 6 个核心字段
- 高级字段可展开但不干扰主流程
- 所有长任务都有进度感知
- 所有高风险动作都有确认与审计
- 履约页面支持图片级防错
- 竞价页面能直观看到 BuyBox、底价和安全区间
- 铺货页面能用看板表达流程
- 不出现前端自造业务状态
