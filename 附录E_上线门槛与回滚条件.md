# 附录 E：上线门槛与回滚条件

## 1. 上线原则

`2026-05-01` 上线不是“功能都能点开”即通过，而是必须满足：

- 账号可控
- 权限可控
- 写操作可审计
- 长任务可追踪
- 外部失败可降级
- 数据可备份
- 事故可回滚

## 2. 上线前硬性门槛

| 类别 | 门槛 | 是否阻断上线 |
|---|---|---:|
| Auth | 登录、登出、`/api/auth/me` 稳定 | 是 |
| Admin | `super_admin` 可管理用户、密码、到期、功能权限 | 是 |
| 权限 | 后端 RBAC 生效，不能只靠前端隐藏 | 是 |
| 审计 | 高危动作有审计日志 | 是 |
| 幂等 | 核心写接口具备幂等保护 | 是 |
| Task | 长任务具备状态、重试、失败原因 | 是 |
| Store | 店铺同步可成功 / 可失败回写 | 是 |
| AutoBid | 无底价禁止竞价，调价有日志 | 是 |
| Fulfillment | 订单池、同仓合单、PO、补号主链跑通 | 是 |
| Finance | 利润快照可读，流水不可覆盖 | 是 |
| Backup | 上线前完成数据库备份 | 是 |
| Rollback | 有明确回滚脚本或步骤 | 是 |
| Monitor | 核心接口、任务、DB、队列可观测 | 是 |
| Security | 登录限流、API 限流、错误脱敏启用 | 是 |

## 3. 功能上线门槛

### `P0` 必须可用

- `Auth / Admin / Subscription`
- `Store Management`
- `AutoBid`
- `Fulfillment / PO Workbench`
- 安全与运行底座

### `P1` 可受限上线

- `Selection`
- `Finance`
- `Warehouse`
- `Extension`

受限上线要求：

- 页面必须明确标注能力边界
- 写能力不足时必须只读
- 不能展示会误导用户的假成功入口

### `P2` 延后补齐

- `Listing / Dropship`

要求：

- `2026-05-01` 可展示任务入口、历史记录、状态壳
- `2026-05-03` ~ `2026-05-05` 补齐完整自动铺货闭环

## 4. 压测门槛

面向 `1000` 用户同时在线的首发要求：

- 登录与 `/api/auth/me` 在压测下稳定
- Dashboard 读取汇总数据，不扫明细热表
- 列表接口分页稳定
- 长任务创建接口可排队削峰
- 外部平台失败不会拖垮主站
- 管理后台和业务接口有独立限流策略

建议验收指标：

| 指标 | 目标 |
|---|---|
| 关键读接口错误率 | `< 1%` |
| 关键写接口重复提交率 | `0` |
| 调价重复执行 | `0` |
| 财务重复入账 | `0` |
| 登录限流生效 | 必须 |
| Worker 失败任务可追踪 | 必须 |

## 5. 上线前检查清单

### 数据

- 已完成上线前数据库备份
- 已确认迁移脚本可重复执行或可检测已执行
- 已确认关键表有必要索引
- 已确认财务流水不可覆盖

### 安全

- 已开启登录限流
- 已开启 API 限流
- 已开启错误脱敏
- 已确认凭证不返回前端明文
- 已确认管理员高危操作二次确认

### 任务

- 已验证任务创建幂等
- 已验证任务失败可重试
- 已验证死信任务可查看
- 已验证取消任务不会留下脏状态

### 外部接口

- 已验证 Takealot 凭证失败场景
- 已验证 Takealot 超时场景
- 已验证 CNExpress 超时场景
- 已验证 1688 / Amazon / AI 失败不会影响 P0 主链

### 前端

- 已验证禁用功能显示原因
- 已验证长任务不是纯 spinner
- 已验证高危动作确认弹层
- 已验证错误反馈可读

## 6. 回滚触发条件

满足任一条件必须进入回滚或降级决策：

- 登录大面积失败
- `/api/auth/me` 不稳定
- `super_admin` 无法进入管理后台
- 出现重复调价
- 出现重复财务入账
- PO 绑定关系错乱
- 数据库迁移导致核心表不可读写
- 外部平台异常导致主站大面积阻塞
- 5xx 错误持续升高
- 队列积压持续增长且无法消费
- 监控与日志不可用

## 7. 降级策略

优先降级，不优先整站回滚。

降级顺序：

1. 关闭 `Listing / Dropship` 自动任务
2. 暂停大规模抓取
3. 暂停全店自动竞价，仅保留查看
4. 暂停外部同步，只保留核心读能力
5. 管理后台保留 `super_admin` 登录与应急开关
6. 必要时进入整站维护模式

## 8. 回滚策略

### 8.1 应用回滚

- 优先使用版本回滚或蓝绿切换
- 保留数据库向前兼容
- 回滚前暂停 worker
- 回滚后执行 smoke test

### 8.2 数据回滚

数据回滚必须非常谨慎。

原则：

- 财务流水不做物理回滚，使用 adjustment 修正
- 调价错误优先通过反向调价修正
- PO 错绑优先通过受审计的人工修正
- 数据库整体恢复只能作为最后手段

### 8.3 Worker 回滚

- 先暂停 worker
- 再停止新任务入队
- 保留队列与 dead letter
- 修复后从安全 checkpoint 恢复

## 9. 上线当天操作顺序

```text
冻结代码与配置
  → 数据库备份
  → 部署后端
  → 部署 worker
  → 部署前端
  → smoke test
  → 开启 super_admin
  → 开启 Auth / Store / Fulfillment
  → 开启 AutoBid 只读
  → 小范围开启 AutoBid 写能力
  → 观察监控
  → 扩大开放
```

- 登录失败率
- `/api/auth/me` 错误率
- 店铺同步成功率
- 任务死信数量

- PO 创建与绑定异常
- 外部 API 超时数量
- 数据库连接池使用率
- 队列积压
- 管理员高危操作审计完整性
